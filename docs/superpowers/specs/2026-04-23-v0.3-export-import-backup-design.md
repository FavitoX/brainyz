# v0.3 — Export / Import / Backup / Restore — Design Spec

**Status**: Approved design (pending spec review and user sign-off)
**Target release**: `v0.3.0`
**Author**: Favio Leyva (with Claude Opus 4.7)
**Date**: 2026-04-23
**Supersedes parts of**: D-016 (Import / Export) — formalizes what was "a definir" in the original decision.

---

## 1. Motivation and scope

The v0.2 product ships a usable CLI + MCP pair built on a local LibSQL database. Users who start accumulating meaningful content have no first-class path for:

1. **Backing up** their brain before risky operations (manual edits, experiments, crash recovery).
2. **Migrating** the brain between machines.
3. **Recovering lost entries** from an older snapshot without overwriting newer ones.
4. **Inspecting or diffing** the brain's contents in a text-friendly format.

D-016 ("Import / Export") sketched export/import for v0.1 but left import semantics and format unresolved. v0.3 formalizes those, introduces the orthogonal `backup`/`restore` pair, and reserves `merge` for the sync work (D-018) that comes later.

### In scope for v0.3

- `brainz export` — JSONL data dump (optionally filtered by project).
- `brainz import` — JSONL ingestion with `replace` or `additive` mode.
- `brainz backup` — atomic `.zip` snapshot of the DB + manifest.
- `brainz restore` — unpack + swap a `.zip` backup, with pre-restore safety backup.
- Unified error-code system (`BZ_<CATEGORY>_<NAME>`) used by all four commands.
- `export` and `backup` exposed as MCP tools; `import` and `restore` are **not**.
- Smoke-test extension so the AOT binary validates backup + restore per-RID in release CI.

### Out of scope for v0.3

- `import --mode merge` (deferred to v0.4+ together with D-018 sync).
- `recovery` command for partial recovery from corrupt JSONL (explicit user note — v0.4+).
- Selective export beyond `--project` (date, tag, query) — deferred.
- Encrypted backup / export — deferred.
- Streaming to stdin/stdout — deferred.
- Schema migration on version mismatch — deferred until a migration system exists.

## 2. Command surface

```
brainz export  --to <path.jsonl>     [--project <id>] [--include-history] [--db <path>]
brainz import  --from <path.jsonl>   [--mode replace|additive] [--no-backup] [--dry-run] [--db <path>]
brainz backup  --to <path.zip>       [--compression fastest|optimal|none] [--db <path>]
brainz restore --from <path.zip>     [--no-backup] [--force] [--db <path>]
```

| Command | Payload format | Layer | Primary use case |
|---|---|---|---|
| `export` | JSONL | data | Inspection, `additive` recovery, git-diffable brain snapshot, interchange |
| `import` | JSONL | data | Recover entries (`additive`) or rebuild DB from JSONL (`replace`) |
| `backup` | `.zip` (`.db` + manifest) | file | Atomic snapshot, fast, migration wholesale |
| `restore` | `.zip` | file | Full recovery, machine migration, rollback |

### Safety invariants (apply to all destructive operations)

1. **Automatic pre-operation backup**: before `import --mode replace` or `restore` touches the local DB, a `brainyz.pre-<op>-YYYYMMDD-HHMMSS.db` snapshot is written to the same directory as the local DB via `VACUUM INTO`. Suppressible with `--no-backup` (intended for CI/tests).
2. **Read-only operations leave the DB untouched**: `export` and `backup` never write to the local DB.
3. **Transactional destructive operations**: `import` runs inside a single transaction; failure at any point yields a `ROLLBACK` and the local DB remains unchanged.
4. **TTY confirmation for `restore`**: when stdin is a TTY, `restore` prints a summary of the operation and waits for `y/N`. Bypass with `--force` or non-TTY stdin (pipe/CI).

### MCP exposure

| Command | Exposed as MCP tool? | Rationale |
|---|---|---|
| `export` | Yes | Read-only from the DB's perspective; safe for an agent to trigger on user request. |
| `backup` | Yes | Read-only; legitimate for an agent to request "snapshot before this refactor". |
| `import` | No | Destructive; should not be triggered by an agent. |
| `restore` | No | Destructive; same rationale. |

## 3. JSONL format specification

### 3.1 Layout

- UTF-8, no BOM.
- Newline-delimited JSON (`\n` between records; `\r\n` accepted on read).
- First line MUST be `{"type":"manifest",...}`.
- Keys use `snake_case` to mirror DB column names — enables direct `jq` inspection and avoids translation when reading alongside schema.sql.
- Each line is a self-contained JSON object; no cross-line references.

### 3.2 Record ordering (required for FK consistency)

1. `manifest` (exactly one, first line)
2. `project`
3. `project_remote`
4. `decision`, `principle`, `note` (core entities)
5. `alternative` (FK → decision)
6. `tag`
7. `tag_link` (M:N entity ↔ tag)
8. `link` (polymorphic cross-entity, D-026)
9. `embedding` (1 per (entity, model))
10. `history_entry` — OFF by default, included only with `--include-history`

### 3.3 Record shapes

**Manifest** (first line only):

```json
{
  "type": "manifest",
  "format_version": 1,
  "brainyz_version": "0.3.0",
  "schema_version": 1,
  "exported_at_ms": 1729680000000,
  "project_filter": null,
  "include_history": false,
  "counts": {
    "project": 3, "project_remote": 3, "decision": 27, "principle": 5,
    "note": 12, "alternative": 18, "tag": 8, "tag_link": 41,
    "link": 9, "embedding": 49, "history_entry": 0
  }
}
```

**Entity records** (one key per column of the underlying table, exact snake_case match). Example for `decision`:

```json
{
  "type": "decision",
  "id": "01JKXN...",
  "project_id": "01JKXK...",
  "title": "LibSQL motor",
  "status": "accepted",
  "confidence": "high",
  "context": "...",
  "decision": "...",
  "rationale": "...",
  "consequences": null,
  "revisit_at_ms": null,
  "created_at_ms": 1729000000000,
  "updated_at_ms": 1729000000000
}
```

**Embedding** — vector is **base64-encoded from the raw `F32_BLOB` bytes** in little-endian IEEE-754 binary32 order (matching LibSQL's `F32_BLOB` and `sqlite-vec` on-disk convention), not a JSON array of floats:

```json
{
  "type": "embedding",
  "entity_type": "decision",
  "entity_id": "01JKXN...",
  "model": "nomic-embed-text:v1.5",
  "dim": 768,
  "content_hash": "sha256:...",
  "vector_base64": "AAAA...",
  "created_at_ms": 1729000000000
}
```

Rationale: a JSON array of 768 floats averages ~8 KB per record in string form; base64 of the same 4-byte-per-float blob is ~4 KB, bit-preserving, parses in constant time. Given embeddings typically account for the majority of an export's size, this roughly halves the file size.

**Link** (polymorphic, D-026):

```json
{
  "type": "link",
  "id": "01JKXR...",
  "from_type": "decision",
  "from_id": "01JKXN...",
  "to_type": "note",
  "to_id": "01JKXQ...",
  "relation_type": "informed_by",
  "created_at_ms": 1729000000000
}
```

**History entries** (only when `--include-history`):

```json
{
  "type": "history_entry",
  "entity_type": "decision",
  "entity_id": "01JKXN...",
  "change_type": "update",
  "changed_at_ms": 1729000000000,
  "snapshot": { /* full row as it was before the change */ }
}
```

### 3.4 Large-file and streaming behavior

- The importer parses the JSONL line-by-line and never materializes the whole file in memory. A 1 GB export (roughly 100k entries with embeddings) stays under a few MB of RSS.
- The exporter streams DB → disk via a writer, one record at a time; memory footprint is `O(1)` in the number of rows.
- No progress reporting in v0.3 (exports and imports complete in seconds to low minutes at realistic sizes). Added in a later release if user feedback says it's missed.

### 3.5 What is NOT exported

- FTS5 virtual tables (`decisions_fts`, etc.) — derived data, rebuilt on import.
- SQLite/LibSQL internal metadata (`sqlite_sequence`, etc.).
- `_meta` table contents — rebuilt from the manifest's `schema_version`.

### 3.6 Forward compatibility

- `format_version: 1` is the v0.3 contract. Incompatible changes → bump to `2`, and future importers will support reading both.
- Unknown record `type` values → log a warning and skip (forward-compat for consumers reading a newer file).
- Extra keys on a record → silently ignored at parse time (forward-compat implicit).

## 4. `import` semantics

### 4.1 Mode `replace` (default)

1. Validate `manifest` (first line of the JSONL):
   - File not found → `BZ_IMPORT_FILE_NOT_FOUND`.
   - First line is missing or not an object → `BZ_IMPORT_MANIFEST_MISSING`.
   - First line is not valid JSON or missing required fields → `BZ_IMPORT_MANIFEST_INVALID_JSON`.
   - `format_version > 1` → `BZ_IMPORT_FORMAT_VERSION_NEWER` (clear forward-incompat message).
   - `format_version == 1` but malformed (e.g. `schema_version` missing or non-integer) → `BZ_IMPORT_MANIFEST_INVALID_JSON`.
   - `schema_version` does not match local → `BZ_IMPORT_SCHEMA_MISMATCH`.
2. **Safety backup**: `VACUUM INTO <db-dir>/brainyz.pre-import-<timestamp>.db`. Skip only on `--no-backup`. Failure → `BZ_IMPORT_SAFETY_BACKUP_FAILED`; operation aborts before any mutation.
3. Begin transaction.
4. `DELETE FROM` each table in the **exact reverse of §3.2 insert order**: `history_entry` (if present), `embedding`, `link`, `tag_link`, `tag`, `alternative`, `note`, `principle`, `decision`, `project_remote`, `project`. FTS5 virtual tables are not deleted row-by-row — after step 6, the FTS tables are dropped and recreated in step 7 to avoid the cost of trigger-driven per-row FTS deletes during step 4. (Disk-full → `BZ_IMPORT_TRANSACTION_FAILED`.)
5. Stream-parse the JSONL, routing each record to its table INSERT in the order guaranteed by §3.2.
   - Record not valid JSON or missing required fields → `BZ_IMPORT_RECORD_MALFORMED` (error message includes line number).
   - INSERT fails a FK constraint → `BZ_IMPORT_FK_VIOLATION` (error message includes the offending record).
   - Any other INSERT failure (disk, unique, CHECK) → `BZ_IMPORT_TRANSACTION_FAILED`.
   - Unknown `type` values → warning logged, record skipped (§3.6 forward-compat).
6. Commit. If the commit itself fails → `BZ_IMPORT_TRANSACTION_FAILED`; rollback is automatic.
7. Drop and recreate FTS5 virtual tables, then backfill with `INSERT INTO <table>_fts SELECT ...`. This is a separate, smaller transaction; failure here leaves data intact and the user can run `brainz reindex` manually — we log a warning and continue with a non-zero exit code only if the replace-mode integrity check (§4.4) also fails.
8. Report: `imported N decisions, M principles, K notes from <path>`.

### 4.2 Mode `additive`

1. Validate `manifest` identically to §4.1 step 1 (same error codes apply).
2. **Safety backup** (also automatic; same rationale — additive can still surface a bug that corrupts local state). Failure → `BZ_IMPORT_SAFETY_BACKUP_FAILED`.
3. Begin transaction.
4. Stream-parse the JSONL. For each record, `INSERT OR IGNORE` into its table:
   - Entities (`decision`, `principle`, `note`): ULID collision → skip silently.
   - Embeddings: PK `(entity_type, entity_id, model)` collision → skip.
   - Links, tag_links: composite unique tuple collision → skip.
   - Record malformed → `BZ_IMPORT_RECORD_MALFORMED` (line number included).
   - Any other INSERT failure → `BZ_IMPORT_TRANSACTION_FAILED`.
5. No `DELETE`s. Nothing local is overwritten.
6. Commit. Failure → `BZ_IMPORT_TRANSACTION_FAILED`.
7. Rebuild FTS5 (full rebuild is simplest and cheap; surgical rebuild deferred unless it becomes a bottleneck).
8. Report: `added N decisions (M already present, skipped), added K embeddings, ...`.

### 4.3 Transaction boundary and atomicity

- Both modes: the entire DB mutation is a single transaction. Any parse/INSERT failure triggers `ROLLBACK`. Local DB is unchanged; the pre-import backup remains on disk for manual review.
- FTS5 rebuild is a separate transaction and is best-effort.
- `--dry-run`: runs §4.x steps 1 and 5 (parse + validate) but never opens a transaction on the local DB. Useful for checking a JSONL before an irreversible operation.

### 4.4 Post-import integrity checks

Run after commit (both modes). Checks and their consequences:

- **Replace mode — row counts per table match the manifest's `counts`.** A mismatch means the DB silently diverged from the authoritative input. Treated as a warning **and** the exit code is bumped to `1`, so CI and scripts detect the divergence. The report explicitly calls out which table mismatched.
- **All embeddings point to an existing entity.** Orphans (entity gone but embedding kept, or vice versa) → warning only, no exit-code bump — these are recoverable with `brainz reindex`.
- **All polymorphic `link` rows have valid `from` and `to` targets.** Cross-entity link to missing target → warning only.

Warnings are always printed at the end of the import summary, even when they do not bump the exit code, so the user sees what drifted.

### 4.5 Embedding model interaction

Embeddings are imported verbatim with their `model` identifier. If the local brain is configured for a different model, queries against the new model will not see imported embeddings (schema-level filter by `model` column per D-011). The user can run `brainz reindex --model <current>` afterward to regenerate.

## 5. `backup` / `restore`

### 5.1 ZIP layout

```
brainyz-backup-YYYYMMDD-HHMMSS.zip
├── manifest.json       # Backup metadata (see 5.2)
├── brainyz.db          # VACUUM INTO snapshot of the local DB
└── README.txt          # Human-readable restore instructions
```

### 5.2 `manifest.json`

```json
{
  "format_version": 1,
  "brainyz_version": "0.3.0",
  "schema_version": 1,
  "created_at_ms": 1729680000000,
  "source_db_path": "C:\\Users\\favit\\AppData\\Roaming\\brainyz\\brainyz.db",
  "source_hostname": "favit-desktop",
  "db_size_bytes": 2457600,
  "embedding_model": "nomic-embed-text:v1.5",
  "counts": {
    "decisions": 27, "principles": 5, "notes": 12, "embeddings": 49
  }
}
```

### 5.3 `backup` flow

1. Validate the local DB exists and is openable. Missing → `BZ_BACKUP_DB_NOT_FOUND`.
2. Temp directory: `<system-temp>/brainyz-backup-<random>/`.
3. `VACUUM INTO '<temp>/brainyz.db'` — atomic at SQLite level, defragmented. Failure (corrupt DB, disk full) → `BZ_BACKUP_VACUUM_FAILED` (disk full specifically → `BZ_BACKUP_DISK_FULL`).
4. Write `manifest.json` with metadata.
5. Write `README.txt` (templated).
6. Create the ZIP at `--to` path. **Default compression is `fastest`** — SQLite pages are moderately entropy-rich (text + embeddings), and `optimal` saves only 5–15% at 3–5× the time on realistic brains; `fastest` is the better default. `--compression optimal` or `--compression none` available for users who prefer smaller archives or inspectable `.db` inside. ZIP write failure (disk full, path invalid) → `BZ_BACKUP_ZIP_WRITE_FAILED` (disk full → `BZ_BACKUP_DISK_FULL`).
7. Clean up temp.
8. Report: `backed up N decisions, M principles to <zip> (<size>)`.

### 5.4 `restore` flow

1. Validate the ZIP:
   - File not found → `BZ_RESTORE_ZIP_NOT_FOUND`.
   - Not a valid ZIP archive → `BZ_RESTORE_ZIP_CORRUPT`.
   - Missing `manifest.json` entry → `BZ_RESTORE_ZIP_MISSING_MANIFEST`.
   - Missing `brainyz.db` entry → `BZ_RESTORE_ZIP_MISSING_DB`.
2. Parse `manifest.json`:
   - `format_version > 1` → `BZ_RESTORE_FORMAT_VERSION_NEWER`.
   - `schema_version` does not match local → `BZ_RESTORE_SCHEMA_MISMATCH`.
3. **Safety backup** of current local DB (if it exists): `VACUUM INTO <db-dir>/brainyz.pre-restore-<timestamp>.db`. Skip only on `--no-backup`. Failure → `BZ_RESTORE_SAFETY_BACKUP_FAILED`; operation aborts before any mutation.
4. Extract `brainyz.db` from the ZIP to a temp path on the **same filesystem** as the target (so step 7 is an atomic rename, not a cross-device copy). Extraction failure (disk full, permission) → `BZ_RESTORE_SWAP_FAILED` (disk full specifically → `BZ_RESTORE_DISK_FULL`).
5. Close any open LibSQL connection held by this process to the local DB (defensive — the CLI opens/closes per command, but this avoids surprises in long-running hosts).
6. **Probe the target DB for external locks** (another `brainz` process or an external tool holding it open). On Windows, `File.Move` over an open target fails; on POSIX the move succeeds but the other process may see inconsistent reads. We detect this by attempting to open the target with an exclusive lock for a brief moment (`FileShare.None` on Windows, `flock(LOCK_EX | LOCK_NB)` on POSIX). If held by another process → `BZ_RESTORE_DB_LOCKED` with a message naming the target path and suggesting the user close any other `brainz mcp` / `brainz <cmd>` instances. No fallback to copy-then-delete (that would trade atomicity for an unsafe window).
7. Atomically **move** (not copy) the temp file to the local DB path. Move failure → `BZ_RESTORE_SWAP_FAILED`.
8. Clean up temp.
9. Report: `restored from <zip> (original from <hostname> on <date>). Pre-restore state saved to <path>`.

**User decline** on the interactive prompt (§5.5) → `BZ_RESTORE_USER_DECLINED` with exit code `3`; no mutation, no safety backup written.

### 5.5 Interactive confirmation (restore only)

When stdin is a TTY **and** `--force` is not passed, print a diff summary and wait for `y/N`:

```
Restoring backup from 2026-04-15 (favit-desktop, schema v1)
This will REPLACE your current brainyz.db at C:\Users\favit\AppData\Roaming\brainyz\brainyz.db
  Current DB: 27 decisions, 5 principles, 12 notes (2.4 MB)
  Backup:     33 decisions, 8 principles, 14 notes (3.1 MB)
A pre-restore backup will be saved to: <path>.pre-restore-<timestamp>.db
Proceed? [y/N]
```

On pipe/CI (non-TTY stdin) the prompt is skipped and the operation proceeds. `--force` forces proceed regardless.

### 5.6 Choosing between `restore` and `import --mode replace`

Both land at "replace local DB". The user picks based on what artifact they have:

| Criterion | `restore --from .zip` | `import --from .jsonl --mode replace` |
|---|---|---|
| Payload | Binary `.db` inside zip | Text JSONL |
| Speed | Near-instant (file swap) | Seconds to minutes (parse + INSERT) |
| FTS5 index | Preserved | Rebuilt post-import |
| History | Always included | Only if exported with `--include-history` |
| Inspectable beforehand | No (binary) | Yes (text, `jq`, git diff) |

Both are legitimate; the CLI does not prefer one.

### 6.4 Dependencies

**None added.** Everything uses the BCL:

- `System.IO.Compression.ZipArchive` — AOT-safe zip handling.
- `System.Text.Json` with source generators (via existing `BrainyzJsonContext`) — no runtime reflection.
- `System.Security.Cryptography.SHA256` — content hashes.
- `Convert.ToBase64String` / `FromBase64String` — embedding encoding.
- `VACUUM INTO` — native LibSQL/SQLite, no driver feature needed.

Net new NuGet count: **0**.

**AOT-safety note on `ZipArchive`**: `System.IO.Compression.ZipArchive` historically emitted trim warnings under aggressive IL trimming for the `Deflate64` code path. We neither read nor write Deflate64 (default `CompressionLevel.Fastest`/`Optimal` use classic Deflate), but to guard against an unexpected AOT regression we rely on the release smoke-test (§8.4) exercising backup + restore against the actual AOT binary for each RID. If a trim or AOT warning appears during `dotnet publish`, the fallback is the same pattern used for the Nelknet native-library workaround (see `src/Brainyz.Cli/NativeLibraryResolver.cs` and commits df741e8 / e21b06c / b1d39d3): a targeted `[DynamicDependency]` or `TrimmerRootDescriptor` in the csproj. No other runtime workaround is required.

### 7.3 User-facing output

```
Error [BZ_IMPORT_SCHEMA_MISMATCH]: Cannot import — export was created for schema version 2, local DB is on version 1.
  Tip: Migration is not yet supported. Re-export from an installation matching the local schema version.
  See: https://github.com/FavitoX/brainyz/blob/master/docs/errors.md#bz_import_schema_mismatch
```

### 7.4 Exit codes

Follows a Unix-flavored convention:

- `0` — success.
- `1` — operation failure (disk full, DB unopenable, transaction rollback).
- `2` — validation failure (bad flags, malformed input, schema mismatch, file not found).
- `3` — user declined an interactive prompt (`BZ_RESTORE_USER_DECLINED`).

### 7.5 Documentation

A new file `docs/errors.md` lists every code, what it means, and recovery guidance. Links from error output to the file's anchors.

## 8. Testing strategy

### 8.1 Unit tests (`Brainyz.Tests`)

- `JsonlExporterTests`: topological order is respected; record shape matches spec; embedding base64 round-trips to original bytes.
- `JsonlImporterTests`: manifest validation; replace vs additive semantics; rollback on parse error; rollback on FK violation; stream parsing does not load full file.
- `ZipBackupTests`: zip structure, manifest shape, VACUUM INTO output is a valid DB, compression modes.
- `ZipRestoreTests`: manifest validation; safety backup is created; atomic swap does not leave temp files; `--force` bypasses prompt correctly; non-TTY stdin skips prompt.

### 8.2 Round-trip tests (the high-value ones)

Seed a DB with a representative mix (projects, entities of every type, alternatives, tags, tag_links, polymorphic links, embeddings at two dimensions, history entries). Then for each combination:

- `export` → `import replace` into a fresh DB → assert equivalent (every row, every column, checksum of embeddings).
- `export --project X` → `import replace` → assert only that project's entries landed plus their relevant refs.
- `export --include-history` → `import replace` → assert history preserved.
- `backup` → `restore` into a fresh DB → same assertions.
- Populated DB + `import additive` with overlap → assert existing rows untouched, new rows added.

### 8.3 Integration / end-to-end tests

Drive the real CLI against a temp DB:

- `brainz init && brainz add decision ... && brainz backup --to X.zip && rm DB && brainz restore --from X.zip && brainz show <id>` — confirms full survival.
- `brainz export --to X.jsonl` followed by `jq` assertions on the file.
- `brainz import --from X.jsonl --mode additive` into a partially-populated DB.
- `brainz import --from X.jsonl --dry-run` — asserts no DB mutation.

### 8.4 Smoke-test extension (release.yml)

The `tools/smoke-test.cs` script gains two new steps appended to the existing nine:

- Step 10: `brainz backup --to $TMP/bkp.zip` → assert zip exists and parses.
- Step 11: `brainz restore --from $TMP/bkp.zip` → assert the previously-added entry survives and can be `show`n.

These run against the just-published AOT binary per RID in `release.yml`, confirming that `ZipArchive`, `System.Text.Json` source-gen, and VACUUM INTO work end-to-end in NativeAOT.

## 10. Open questions / deferred

These are **not** being designed now but are tracked so they don't get lost:

- **`merge` mode** (import): belongs with D-018 sync. Conflict resolution by timestamp + user intervention.
- **`recovery` command**: partial recovery from corrupt JSONL. v0.4+ if there's demand.
- **Selective export beyond `--project`**: by date range, tag, query.
- **Encrypted backup/export**: passphrase-derived key for the zip.
- **Streaming to stdin/stdout**: useful for piping into other tools.
- **Schema migration**: when `_meta.schema_version` evolves, import/restore need migration paths.
- **Tag `normalized_name` collision in `additive`**: when two tags share a `normalized_name` but differ in `name`/`path`, additive import skips silently by ULID (which is correct per the INSERT OR IGNORE policy) but the user may expect a merge. Formalize the behavior when tag normalization is revisited.

## 11. Summary

v0.3 introduces two orthogonal pairs of commands — `export`/`import` (data-level, JSONL, text-friendly) and `backup`/`restore` (file-level, ZIP, fast) — that together cover the recovery, migration, and inspection use cases without stepping into the conflict-resolution territory reserved for D-018 sync. The design adds zero runtime dependencies, extends the existing error-code discipline with a unified prefix, and extends the release smoke-test so AOT regressions on the new code paths are caught before a release is cut.
