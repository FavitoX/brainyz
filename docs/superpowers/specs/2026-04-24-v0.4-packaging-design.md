# v0.4 — Packaging extendido — Design Spec

**Status**: Approved design (pending spec review and user sign-off)
**Target release**: `v0.4.0`
**Author**: Favio Leyva (with Claude Opus 4.7)
**Date**: 2026-04-24
**Supersedes parts of**: D-021 (Distribución y packaging) — formalizes the v0.2/v0.3 entries that were deferred and consolidates the remaining channels into a single v0.4 ship.

---

## 1. Motivation and scope

v0.3 shipped a fully-featured CLI + MCP binary distributed as tar.gz / zip archives on GitHub Releases. Users have to:

1. Know the right archive for their OS/arch.
2. Download, extract, place the binary on `PATH` manually.
3. Re-do all of the above on every update.

That friction is the single largest adoption tax left. Every mature OSS devtool (ripgrep, bat, fd, zoxide, gh, starship) is installable with one command via the platform-native package manager; brainyz is not. D-021 sketched a staged distribution roadmap but the v0.2 and v0.3 packaging entries never shipped — this spec consolidates them into v0.4 and designs them to be automated from the existing release pipeline wherever it pays off.

### In scope for v0.4

- **Homebrew tap** (macOS + Linux) — `brew install chaosfabric/tap/brainyz`.
- **Scoop bucket** (Windows) — `scoop bucket add chaosfabric … && scoop install brainyz`.
- **Winget** (Windows, Microsoft Store / WPM) — `winget install ChaosFabric.Brainyz`.
- **AUR `-bin`** (Arch Linux) — `yay -S brainyz-bin`.
- **SHA256 checksums** published alongside the release archives (precondition for all four channels).
- **Release-workflow automation** for Homebrew tap and Scoop bucket (push to org-owned repos on stable tags).
- **Manual bootstrap** for Winget and AUR on v0.4.0, with the templates versioned in org-owned shadow repos so later automation has a starting point.
- **Install docs** (`README.md` "Install" section + new `docs/packaging.md`).
- **`bz` alias** installed alongside `brainz` by every channel (D-019 mitigation for the `brainz` binary-name collision with Varnish Controller Brainz and the Brainease/BrainZ esolangs).

### Out of scope for v0.4

- **Code signing**: Microsoft Authenticode + Apple Notarization. D-021 explicitly defers both until the project has traction; Winget and Homebrew accept unsigned binaries.
- **nix flake**: D-021 conditions it on community demand — no open request yet.
- **AUR source-build package** (`brainyz`, non-`-bin`): needs .NET SDK at build time; the `-bin` package installs the pre-built NativeAOT binary directly from GitHub Releases and covers the overwhelming majority of users.
- **Scoop auto-PR / Winget auto-PR in CI**: automation for those two channels is deferred to v0.5+ after the manual bootstrap proves the manifests work. See §5.
- **Repository transfer** of `FavitoX/brainyz` into the new org: the main repo stays at `FavitoX/brainyz`. Only the tap/bucket/shadow repos live under `chaosfabric`.
- **`brainyz` Homebrew core submission** (vs. tap): cores require maturity, a stable release cadence, and review by Homebrew maintainers. Tap is the right scope for v0.4.

## 2. Architecture overview

```
                             ┌────────────────────────────────────┐
 git tag v0.4.0              │  .github/workflows/release.yml     │
 ─────────────────────►      │                                    │
                             │  job: build (×4 RIDs)              │
                             │    → tar.gz / zip artifacts        │
                             │  job: checksums  (new)             │
                             │    → SHA256SUMS.txt                │
                             │  job: release                      │
                             │    → GitHub Release + artifacts    │
                             │  job: update-homebrew-tap  (new)   │
                             │    → push Formula/brainyz.rb       │
                             │      to chaosfabric/homebrew-tap   │
                             │  job: update-scoop-bucket  (new)   │
                             │    → push bucket/brainyz.json      │
                             │      to chaosfabric/scoop-bucket   │
                             └────────────────────────────────────┘

                             Manual-in-v0.4 (automated in v0.5+):

                             wingetcreate  ─► microsoft/winget-pkgs (PR)
                                           └► chaosfabric/winget-manifests (versioned copy)
                             aurpublish    ─► ssh://aur@aur.archlinux.org/brainyz-bin.git
                                           └► chaosfabric/aur-brainyz-bin (versioned copy)
```

### 2.1 Design principles

- **No app code changes.** Everything is CI config, package manifests, and docs. Zero risk of regression in the shipped binary.
- **Single source of truth for version + checksums.** `SHA256SUMS.txt` published with the GitHub Release is the only artifact any downstream package reads. No embedded manifests in the main repo that could drift from the release.
- **Idempotent automation.** Homebrew/Scoop updater jobs are safe to re-run on the same tag (they recompute from the release, overwrite the formula/manifest, force-push-free).
- **Fail-open per channel.** If the Homebrew job fails, the GitHub Release still shipped and the other three channels are untouched. Each updater job is `continue-on-error: true` with clear log output so a single channel outage never gates the release.
- **Prerelease gates stable-only channels.** Tags containing `-` (`v0.4.0-rc.1`, etc.) skip all updater jobs. Package managers ship stable only.

## 3. Release-artifact changes

### 3.1 `SHA256SUMS.txt` (new)

A new `checksums` job runs after `build` and before `release`:

```yaml
checksums:
  name: generate SHA256SUMS
  needs: build
  runs-on: ubuntu-latest
  steps:
    - uses: actions/download-artifact@v8
      with:
        path: artifacts
        merge-multiple: true
    - name: Compute checksums
      run: |
        cd artifacts
        sha256sum brainz-*.tar.gz brainz-*.zip > SHA256SUMS.txt
        cat SHA256SUMS.txt
    - uses: actions/upload-artifact@v7
      with:
        name: checksums
        path: artifacts/SHA256SUMS.txt
```

The existing `release` job downloads it alongside the other artifacts and uploads it to the GitHub Release via `softprops/action-gh-release`. Output shape:

```
brainz-linux-x64.tar.gz
brainz-linux-arm64.tar.gz
brainz-osx-arm64.tar.gz
brainz-win-x64.zip
SHA256SUMS.txt                  ← new
```

### 3.2 Archive contents (unchanged)

Archives keep their current layout; the existing `Package (tar.gz)` step on Linux/macOS already bundles `libsql.{so,dylib}` next to `brainz`, and the `Package (zip)` step on Windows already bundles `brainz.exe` + `libsql.dll`. All four package managers install the unmodified archive contents verbatim.

## 4. Per-channel manifests

### 4.1 Homebrew tap — `chaosfabric/homebrew-tap`

**Layout**:
```
Formula/
  brainyz.rb
README.md         ← one-liner pointing back at FavitoX/brainyz
```

**Formula skeleton** (generated by the updater job, illustrative):

```ruby
# Copyright 2026 Favio Andres Leyva
# SPDX-License-Identifier: Apache-2.0
class Brainyz < Formula
  desc "Global-first knowledge layer with per-project scoping and MCP integration"
  homepage "https://github.com/FavitoX/brainyz"
  version "0.4.0"
  license "Apache-2.0"

  on_macos do
    on_arm do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-osx-arm64.tar.gz"
      sha256 "<sha256-osx-arm64>"
    end
  end

  on_linux do
    on_intel do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-linux-x64.tar.gz"
      sha256 "<sha256-linux-x64>"
    end
    on_arm do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-linux-arm64.tar.gz"
      sha256 "<sha256-linux-arm64>"
    end
  end

  def install
    libexec.install Dir["*"]
    (bin/"brainz").write <<~SHIM
      #!/bin/bash
      exec "#{libexec}/brainz" "$@"
    SHIM
    chmod 0755, bin/"brainz"
    bin.install_symlink bin/"brainz" => "bz"
  end

  test do
    assert_match "brainyz #{version}", shell_output("#{bin}/brainz --version")
  end
end
```

**Native-library handling**: `libsql.{so,dylib}` lives in `libexec/` next to the binary. macOS resolves sibling-dir dylibs via the main binary's `@loader_path` (implicit for AOT single-file output). Linux resolves sibling `.so` files via rpath `$ORIGIN` which NativeAOT sets by default. If either fails on real install, the shim becomes:

```bash
#!/bin/bash
export LD_LIBRARY_PATH="#{libexec}:${LD_LIBRARY_PATH}"      # Linux
export DYLD_LIBRARY_PATH="#{libexec}:${DYLD_LIBRARY_PATH}"  # macOS
exec "#{libexec}/brainz" "$@"
```

Verified by running `brew install` from a clean Homebrew in a CI matrix (see §7.1).

**`bz` alias**: installed as a `bin.install_symlink` pointing at the `brainz` shim. One-liner, universally portable.

**linux-arm64 note**: the archive ships without `libsql.so` (Nelknet.LibSQL.Bindings 0.2.5 does not bundle one — documented in `nelknet-bugs.md` §8). The Homebrew formula ships the binary anyway; `brew install` succeeds and `brainz --version` works. Any command that opens the DB fails with a clear message directing users to supply their own aarch64 `libsql.so`. This matches the non-blocking smoke-test policy already in `release.yml`.

### 4.2 Scoop bucket — `chaosfabric/scoop-bucket`

**Layout**:
```
bucket/
  brainyz.json
README.md
```

**Manifest skeleton**:

```json
{
  "version": "0.4.0",
  "description": "Global-first knowledge layer with per-project scoping and MCP integration",
  "homepage": "https://github.com/FavitoX/brainyz",
  "license": "Apache-2.0",
  "architecture": {
    "64bit": {
      "url": "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-win-x64.zip",
      "hash": "<sha256-win-x64>"
    }
  },
  "bin": [
    "brainz.exe",
    ["brainz.exe", "bz"]
  ],
  "checkver": "github",
  "autoupdate": {
    "architecture": {
      "64bit": {
        "url": "https://github.com/FavitoX/brainyz/releases/download/v$version/brainz-win-x64.zip"
      }
    }
  }
}
```

`"bin"` entries register both `brainz.exe` and a `bz` shim on `PATH`. `libsql.dll` lives alongside `brainz.exe` in the extracted directory; Windows loader finds it next to the executable with no extra config.

`checkver` + `autoupdate` are defensive: our updater job pushes the manifest directly, so autoupdate is not needed in the normal path — but leaving it in means Scoop's own nightly check will keep the bucket in sync if our CI ever stops pushing.

### 4.3 Winget — `chaosfabric/winget-manifests` (shadow) + PR to `microsoft/winget-pkgs`

**Shadow-repo layout** (mirrors the path microsoft/winget-pkgs expects):
```
manifests/c/ChaosFabric/Brainyz/
  0.4.0/
    ChaosFabric.Brainyz.yaml
    ChaosFabric.Brainyz.installer.yaml
    ChaosFabric.Brainyz.locale.en-US.yaml
```

**Manifests generated by `wingetcreate new --urls <url>.zip` locally, then hand-polished**. Key fields:

- `PackageIdentifier: ChaosFabric.Brainyz`
- `Publisher: ChaosFabric`
- `PackageName: Brainyz`
- `License: Apache-2.0`
- `Installers:` entry with `Architecture: x64`, `InstallerType: zip`, `NestedInstallerType: portable`, `NestedInstallerFiles: brainz.exe`, the release-URL and the SHA256.
- `Commands: [brainz, bz]` — registers both on PATH.

**Submission flow on v0.4.0**:
1. `wingetcreate new https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-win-x64.zip` on a Windows box.
2. Answer the prompts; accept the generated SHA256.
3. Commit the three YAMLs to `chaosfabric/winget-manifests`.
4. `wingetcreate submit --token <PAT>` opens the PR to `microsoft/winget-pkgs`.
5. Wait 2–7 days for review + smoke test. Address any lint findings.
6. Once merged, `winget install ChaosFabric.Brainyz` works for everyone.

**Automation deferred to v0.5+**: a CI job that runs `wingetcreate update --urls <v0.5.0-url> --version 0.5.0 ChaosFabric.Brainyz --submit --token $WINGET_PAT` on each stable tag. Prereq is a GitHub PAT with `public_repo` scope stored in secrets; that's a cost we take after the first manual PR proves the manifests.

### 4.4 AUR — `chaosfabric/aur-brainyz-bin` (shadow) + direct push to AUR

**Shadow-repo layout**:
```
PKGBUILD
.SRCINFO
README.md
```

**`PKGBUILD` skeleton**:

```bash
# Maintainer: Favio Leyva <andy.carp@gmail.com>
pkgname=brainyz-bin
_pkgname=brainyz
pkgver=0.4.0
pkgrel=1
pkgdesc="Global-first knowledge layer with per-project scoping and MCP integration"
arch=('x86_64')
url="https://github.com/FavitoX/brainyz"
license=('Apache-2.0')
provides=('brainyz')
conflicts=('brainyz')
source_x86_64=("$pkgname-$pkgver.tar.gz::https://github.com/FavitoX/brainyz/releases/download/v$pkgver/brainz-linux-x64.tar.gz")
sha256sums_x86_64=('<sha256-linux-x64>')
options=(!strip)  # AOT binaries are already stripped

package() {
  install -Dm755 "$srcdir/brainz" "$pkgdir/usr/bin/brainz"
  # libsql.so ships alongside; place in same dir and set rpath via a wrapper
  # if AOT $ORIGIN resolution proves unreliable on Arch glibc.
  if [ -f "$srcdir/libsql.so" ]; then
    install -Dm755 "$srcdir/libsql.so" "$pkgdir/usr/lib/brainyz/libsql.so"
    # Replace /usr/bin/brainz with a shim that sets LD_LIBRARY_PATH.
    mv "$pkgdir/usr/bin/brainz" "$pkgdir/usr/lib/brainyz/brainz"
    install -d "$pkgdir/usr/bin"
    cat > "$pkgdir/usr/bin/brainz" <<'EOF'
#!/bin/sh
export LD_LIBRARY_PATH="/usr/lib/brainyz:${LD_LIBRARY_PATH}"
exec "/usr/lib/brainyz/brainz" "$@"
EOF
    chmod 0755 "$pkgdir/usr/bin/brainz"
  fi
  # bz alias
  ln -s brainz "$pkgdir/usr/bin/bz"
}

check() {
  "$srcdir/brainz" --version | grep -q "brainyz $pkgver"
}
```

**`.SRCINFO` is generated with `makepkg --printsrcinfo > .SRCINFO`**; AUR tooling requires it to be committed alongside the PKGBUILD.

**Submission flow on v0.4.0**:
1. On Arch (WSL / VM / container): register an AUR account, add SSH public key in the web profile.
2. `git clone ssh://aur@aur.archlinux.org/brainyz-bin.git` (first push creates the repo — it starts empty).
3. Copy the PKGBUILD + .SRCINFO in, test locally with `makepkg -si`.
4. `git add PKGBUILD .SRCINFO && git commit -m "v0.4.0" && git push`.
5. Snapshot the same files in `chaosfabric/aur-brainyz-bin` for GitHub-side review/history.

**Automation deferred to v0.5+**: a CI job that takes `pkgver` from the tag, templates PKGBUILD + `.SRCINFO`, then `git push` to AUR over SSH with a deploy key in secrets.

**ARM64 not covered**: `arch=('x86_64')` only. `aarch64` can be added later when Nelknet ships the native library for it (nelknet-bugs.md §8).

## 5. Release-workflow changes

### 5.1 Job topology

Before:
```
build ─► release
```

After:
```
build ─► checksums ─► release ─► update-homebrew-tap (parallel)
                               └► update-scoop-bucket (parallel)
```

### 5.2 `update-homebrew-tap` (new)

```yaml
update-homebrew-tap:
  name: update Homebrew tap
  needs: release
  runs-on: ubuntu-latest
  if: "!contains(github.ref_name, '-')"   # stable tags only
  continue-on-error: true
  steps:
    - name: Compute release metadata
      id: meta
      run: |
        TAG="${{ github.event.inputs.tag || github.ref_name }}"
        VERSION="${TAG#v}"
        echo "tag=$TAG"         >> "$GITHUB_OUTPUT"
        echo "version=$VERSION" >> "$GITHUB_OUTPUT"
    - name: Fetch SHA256SUMS
      run: |
        curl -fsSL -o SHA256SUMS.txt \
          "https://github.com/FavitoX/brainyz/releases/download/${{ steps.meta.outputs.tag }}/SHA256SUMS.txt"
    - name: Render Formula
      run: ./tools/render-homebrew-formula.sh \
             ${{ steps.meta.outputs.version }} \
             ${{ steps.meta.outputs.tag }} \
             SHA256SUMS.txt \
             > Formula-brainyz.rb
    - name: Push to chaosfabric/homebrew-tap
      env:
        DEPLOY_KEY: ${{ secrets.HOMEBREW_TAP_DEPLOY_KEY }}
      run: |
        mkdir -p ~/.ssh && echo "$DEPLOY_KEY" > ~/.ssh/id_ed25519
        chmod 0600 ~/.ssh/id_ed25519
        ssh-keyscan github.com >> ~/.ssh/known_hosts
        git clone git@github.com:chaosfabric/homebrew-tap.git tap
        cp Formula-brainyz.rb tap/Formula/brainyz.rb
        cd tap
        git -c user.name='brainyz-release-bot' \
            -c user.email='releases@chaosfabric.dev' \
            commit -am "brainyz ${{ steps.meta.outputs.version }}"
        git push
```

### 5.3 `update-scoop-bucket` (new)

Mirror of 5.2: renders `bucket/brainyz.json` from a second template script `tools/render-scoop-manifest.sh`, pushes to `chaosfabric/scoop-bucket` via deploy key (`SCOOP_BUCKET_DEPLOY_KEY`).

### 5.4 Template scripts

Lives under `tools/`:
- `tools/render-homebrew-formula.sh` — bash, `sed`-substitutes `{{VERSION}}`, `{{TAG}}`, `{{SHA256_OSX_ARM64}}`, `{{SHA256_LINUX_X64}}`, `{{SHA256_LINUX_ARM64}}` into `tools/templates/homebrew-brainyz.rb.tmpl`.
- `tools/render-scoop-manifest.sh` — same for `tools/templates/scoop-brainyz.json.tmpl`.

Both scripts are thin — ~20 lines each — and testable locally with a hand-written `SHA256SUMS.txt`.

### 5.5 Secrets required (one-time setup)

- `HOMEBREW_TAP_DEPLOY_KEY` — SSH private key with write access to `chaosfabric/homebrew-tap` (ed25519 recommended).
- `SCOOP_BUCKET_DEPLOY_KEY` — same for `chaosfabric/scoop-bucket`.

Deploy keys (vs. PAT) scope access to exactly those two repos; compromise of the key does not grant access to anything else in the org.

## 7. Testing and verification

### 7.1 Automated in CI (new)

- **`SHA256SUMS.txt` validation** — the `release` job verifies the file matches every uploaded artifact before publishing. One-line `sha256sum -c` check.
- **Template-render smoke** — unit test that `tools/render-homebrew-formula.sh` produces a valid Ruby file (`ruby -c`) given a golden input. Same for Scoop (`jq empty`).
- **Homebrew formula lint** — a `brew audit --strict chaosfabric/tap/brainyz` step added to a new `packaging.yml` workflow that runs weekly. Catches deprecated formula DSL, missing test blocks, etc., without gating any individual release.

### 7.2 Manual verification per release

Checklist embedded in `docs/packaging.md`, run once per stable release against a clean machine:

- [ ] macOS arm64: `brew install chaosfabric/tap/brainyz && brainz --version` shows the new tag.
- [ ] Linux x64: same, on a fresh Ubuntu container.
- [ ] Windows: `winget install ChaosFabric.Brainyz && brainz.exe --version`.
- [ ] Windows (Scoop): `scoop install chaosfabric/brainyz && brainz.exe --version`.
- [ ] Arch: `yay -S brainyz-bin && brainz --version` inside a fresh Arch container.
- [ ] On every platform: `bz --version` produces the same output as `brainz --version`.

### 7.3 Smoke-test binary (unchanged)

`tools/smoke-test.cs` keeps running per-RID against the raw published binary. Packaging layers sit on top of that; if the binary is broken, packaging never runs.

### 7.4 Rollback procedure

Per channel, documented in `docs/packaging.md`:

| Channel | Rollback action |
|---|---|
| Homebrew tap | `git revert <bad-commit> && git push` on `chaosfabric/homebrew-tap`. |
| Scoop bucket | `git revert <bad-commit> && git push` on `chaosfabric/scoop-bucket`. |
| Winget | Open a PR to `microsoft/winget-pkgs` removing the offending version's manifest folder. |
| AUR | `git push` the previous PKGBUILD to `ssh://aur@aur.archlinux.org/brainyz-bin.git`. |

The GitHub Release itself is never edited — the underlying archives remain available for manual install at all times. A user who already installed the bad version downgrades with their package manager's standard mechanism.

## 8. Error handling and failure modes

### 8.1 Updater-job failures

Both `update-homebrew-tap` and `update-scoop-bucket` are `continue-on-error: true`. Failure modes and recovery:

- **Deploy key rejected** → check the secret is populated; re-run the job on the same tag (idempotent).
- **SHA256SUMS.txt not yet visible** → GitHub Release propagation delay (rare, <1 min). Re-run the job.
- **Template script produces invalid file** → caught locally by the §7.1 smoke test; if it lands in CI it's an advisory failure, tap/bucket is untouched, fix the template and re-run.

Every failure leaves prior state intact — formula/manifest is not half-written.

### 8.2 Manual-submission failures

- **Winget PR rejected** (validation errors from Microsoft's CI) → fix the manifest in `chaosfabric/winget-manifests`, push a new commit to the PR.
- **Winget SmartScreen warnings** — unsigned binary. Expected per §1 "out of scope"; `docs/packaging.md` includes a note for users that "More info → Run anyway" is the current path until signing lands.
- **AUR build fails** on a user's machine → `makepkg -si` error lines; bug filed against `chaosfabric/aur-brainyz-bin` (the shadow repo) for visibility, fix pushed to AUR directly.

### 8.3 Package manager outages

Each channel failing in isolation is tolerable — users have four paths to install, plus the raw tar.gz/zip. No error-code system needed; these failures surface as HTTP errors or package-manager messages, outside the app boundary.

## 9. Security considerations

- **Checksums are the trust root.** Every channel pins to the SHA256 published in the GitHub Release. An attacker who compromised the tap/bucket repos could only change which release is installed (all real binaries are in FavitoX/brainyz/releases, which has its own auth and audit trail).
- **Deploy keys are single-repo scoped.** Compromise of the Homebrew deploy key does not grant access to the Scoop bucket or the main repo.
- **No secrets flow through manifests.** The manifests are pure metadata + URLs + SHA256; nothing a package-manager user downloads is derived from our secrets.
- **Unsigned binaries** — Microsoft SmartScreen and macOS Gatekeeper will show warnings. Documented in `docs/packaging.md`. Signing is deferred per §1.

## 10. Consequences and follow-ups

### 10.1 Ongoing cost

- Homebrew tap + Scoop bucket: zero manual work per release (CI handles it).
- Winget: ~10 min per release to run `wingetcreate update` + submit, until the automation lands in v0.5+.
- AUR: ~5 min per release to push a new `pkgver`, until automation lands.

Total human-in-the-loop per stable release in v0.4: ~15 minutes; ~0 minutes from v0.5+ once Winget/AUR automation lands.

### 10.2 Observability

A one-liner weekly cron that curls `https://api.github.com/repos/FavitoX/brainyz/releases/latest` vs. each package manager's latest and reports drift. Nice-to-have, tracked in `docs/packaging.md` futures section. Out of scope for v0.4.

### 10.3 When to revisit

- **Code signing** — when SmartScreen friction appears in user reports.
- **nix flake** — when an issue requests it.
- **Homebrew core submission** — after six consecutive stable releases with no regressions.
- **AUR source package** — if a contributor opens a PR (we won't chase it ourselves).

### 10.4 Decision log update

D-021 is marked "implemented for v0.4" for the Homebrew tap, Scoop bucket, Winget, and AUR entries. `nix` and code-signing entries remain explicitly deferred with the triggering conditions above.
