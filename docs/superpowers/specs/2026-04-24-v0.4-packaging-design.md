# v0.4 — Packaging extendido — Design Spec

**Status**: Approved design (pending spec review and user sign-off)
**Target release**: `v0.4.0`
**Author**: Favio Leyva (with Claude Opus 4.7)
**Date**: 2026-04-24
**Supersedes parts of**: D-021 (Distribución y packaging) — formalizes the v0.2/v0.3 entries that were deferred and consolidates the remaining channels into a single v0.4 ship.

---

## 1. Motivation and scope

v0.3 shipped a fully-featured CLI + MCP binary distributed as tar.gz / zip archives on GitHub Releases. Users have to:

1. Know the right archive for their OS/arch.
2. Download, extract, place the binary on `PATH` manually.
3. Re-do all of the above on every update.

That friction is the single largest adoption tax left. Every mature OSS devtool (ripgrep, bat, fd, zoxide, gh, starship) is installable with one command via the platform-native package manager; brainyz is not. D-021 sketched a staged distribution roadmap but the v0.2 and v0.3 packaging entries never shipped — this spec consolidates them into v0.4 and designs them to be automated from the existing release pipeline wherever it pays off.

### In scope for v0.4

- **Homebrew tap** (macOS + Linux) — `brew install chaosfabric/tap/brainyz`.
- **Scoop bucket** (Windows) — `scoop bucket add chaosfabric … && scoop install brainyz`.
- **Winget** (Windows, Microsoft Store / WPM) — `winget install ChaosFabric.Brainyz`.
- **AUR `-bin`** (Arch Linux) — `yay -S brainyz-bin`.
- **SHA256 checksums** published alongside the release archives (precondition for all four channels).
- **Release-workflow automation** for Homebrew tap and Scoop bucket (push to org-owned repos on stable tags).
- **Manual bootstrap** for Winget and AUR on v0.4.0, with the templates versioned in org-owned shadow repos so later automation has a starting point.
- **Install docs** (`README.md` "Install" section + new `docs/packaging.md`).
- **`bz` alias** installed alongside `brainz` by every channel (D-019 mitigation for the `brainz` binary-name collision with Varnish Controller Brainz and the Brainease/BrainZ esolangs).
- **Wire `brainz --version` output to `BrainyzCliVersion.Current`** (small app-code change, see §3.3). Precondition for the Homebrew `test do` and AUR `check()` assertions to pass.
- **Drift-detection workflow** (`packaging.yml`) that runs weekly and opens a GitHub issue if any channel's published version trails the latest GitHub Release (see §7.1). Offsets the "silent failure" cost of `continue-on-error: true` on the updater jobs.

### Out of scope for v0.4

- **Code signing**: Microsoft Authenticode + Apple Notarization. D-021 explicitly defers both until the project has traction; Winget and Homebrew accept unsigned binaries.
- **nix flake**: D-021 conditions it on community demand — no open request yet.
- **AUR source-build package** (`brainyz`, non-`-bin`): needs .NET SDK at build time; the `-bin` package installs the pre-built NativeAOT binary directly from GitHub Releases and covers the overwhelming majority of users.
- **Scoop auto-PR / Winget auto-PR in CI**: automation for those two channels is deferred to v0.5+ after the manual bootstrap proves the manifests work. See §5.
- **Repository transfer** of `FavitoX/brainyz` into the new org: the main repo stays at `FavitoX/brainyz`. Only the tap/bucket/shadow repos live under `chaosfabric`.
- **`brainyz` Homebrew core submission** (vs. tap): cores require maturity, a stable release cadence, and review by Homebrew maintainers. Tap is the right scope for v0.4.

## 2. Architecture overview

```
                             ┌────────────────────────────────────┐
 git tag v0.4.0              │  .github/workflows/release.yml     │
 ─────────────────────►      │                                    │
                             │  job: build (×4 RIDs)              │
                             │    → tar.gz / zip artifacts        │
                             │  job: checksums  (new)             │
                             │    → SHA256SUMS.txt                │
                             │  job: release                      │
                             │    → GitHub Release + artifacts    │
                             │  job: update-homebrew-tap  (new)   │
                             │    → push Formula/brainyz.rb       │
                             │      to chaosfabric/homebrew-tap   │
                             │  job: update-scoop-bucket  (new)   │
                             │    → push bucket/brainyz.json      │
                             │      to chaosfabric/scoop-bucket   │
                             └────────────────────────────────────┘

                             Manual-in-v0.4 (automated in v0.5+):

                             wingetcreate  ─► microsoft/winget-pkgs (PR)
                                           └► chaosfabric/winget-manifests (versioned copy)
                             aurpublish    ─► ssh://aur@aur.archlinux.org/brainyz-bin.git
                                           └► chaosfabric/aur-brainyz-bin (versioned copy)
```

### 2.1 Design principles

- **Minimal app-code footprint.** One localized change — making `brainz --version` print `brainyz <ver>` sourced from `BrainyzCliVersion.Current` (§3.3). Everything else is CI config, package manifests, and docs. The app-code change is testable in isolation and does not touch storage, search, or any command semantics.
- **Single source of truth for version + checksums.** `SHA256SUMS.txt` published with the GitHub Release is the only artifact any downstream package reads. No embedded manifests in the main repo that could drift from the release.
- **Idempotent automation.** Homebrew/Scoop updater jobs are safe to re-run on the same tag (they recompute from the release, overwrite the formula/manifest, force-push-free).
- **Fail-open per channel, but alert on drift.** If the Homebrew job fails, the GitHub Release still shipped and the other three channels are untouched. Each updater job is `continue-on-error: true` with clear log output so a single channel outage never gates the release. A separate weekly `packaging.yml` job detects channel-vs-release drift and opens a GitHub issue — the "silent failure" backstop.
- **Prerelease gates stable-only channels.** Tags containing `-` (`v0.4.0-rc.1`, etc.) skip all updater jobs. Package managers ship stable only. The gate checks both `github.ref_name` (tag push) and `github.event.inputs.tag` (`workflow_dispatch`) so manual re-runs behave identically to tag pushes.

## 3. Release-artifact changes

### 3.1 `SHA256SUMS.txt` (new)

A new `checksums` job runs after `build` and before `release`:

```yaml
checksums:
  name: generate SHA256SUMS
  needs: build
  runs-on: ubuntu-latest
  steps:
    - uses: actions/download-artifact@v8
      with:
        path: artifacts
        merge-multiple: true
    - name: Compute checksums
      run: |
        cd artifacts
        sha256sum brainz-*.tar.gz brainz-*.zip > SHA256SUMS.txt
        cat SHA256SUMS.txt
    - uses: actions/upload-artifact@v7
      with:
        name: checksums
        path: artifacts/SHA256SUMS.txt
```

The existing `release` job downloads it alongside the other artifacts and uploads it to the GitHub Release via `softprops/action-gh-release`. Output shape:

```
brainz-linux-x64.tar.gz
brainz-linux-arm64.tar.gz
brainz-osx-arm64.tar.gz
brainz-win-x64.zip
SHA256SUMS.txt                  ← new
```

### 3.2 Archive contents (unchanged)

Archives keep their current layout; the existing `Package (tar.gz)` step on Linux/macOS already bundles `libsql.{so,dylib}` next to `brainz`, and the `Package (zip)` step on Windows already bundles `brainz.exe` + `libsql.dll`. All four package managers install the unmodified archive contents verbatim.

### 3.3 `brainz --version` output (new, small app-code change)

**Current state** (reason for the change): `Brainyz.Cli.csproj` has no `<Version>` property → assembly version defaults to `1.0.0.0`. `Program.cs` builds a bare `RootCommand` with no custom `--version` handler, so `System.CommandLine` 2.0.7 prints the assembly version (`1.0.0.0`). `BrainyzCliVersion.Current` (`const string = "0.3.0"`) is already the single source of truth for the export/backup manifest version — but it is not wired into CLI `--version`.

**Change**:

1. In `Brainyz.Cli.csproj`, add `<Version>$(BrainyzCliVersion)</Version>` plus a small MSBuild target `GetVersionFromSource` that reads `BrainyzCliVersion.cs` via regex into `$(BrainyzCliVersion)` so the MSBuild property and the C# constant stay in lockstep (single source of truth remains `BrainyzCliVersion.cs`). Target outline:
   ```xml
   <Target Name="GetVersionFromSource" BeforeTargets="BeforeBuild">
     <PropertyGroup>
       <_VersionSource>$([System.IO.File]::ReadAllText('$(MSBuildThisFileDirectory)BrainyzCliVersion.cs'))</_VersionSource>
       <BrainyzCliVersion>$([System.Text.RegularExpressions.Regex]::Match($(_VersionSource), 'Current\s*=\s*"([^"]+)"').Groups[1].Value)</BrainyzCliVersion>
     </PropertyGroup>
   </Target>
   ```

2. In `Program.cs`, override the built-in `VersionOption.Action`. Rationale: `RootCommand` in System.CommandLine 2.0.0-beta5+ (which 2.0.7 is) auto-attaches a `VersionOption` whose `Action` intercepts `--version` *before* any `root.SetAction` runs — a plain `root.SetAction(... GetValue<bool>("--version") ...)` would never fire. The correct shape:
   ```csharp
   using System.CommandLine;
   using System.CommandLine.Invocation;

   var versionOption = root.Options.OfType<VersionOption>().Single();
   versionOption.Action = new PrintVersionAction();

   // ...

   internal sealed class PrintVersionAction : SynchronousCommandLineAction
   {
       public PrintVersionAction() { ClearsParseErrors = true; }
       public override int Invoke(ParseResult parseResult)
       {
           Console.WriteLine($"brainyz {BrainyzCliVersion.Current}");
           return 0;
       }
   }
   ```
   (References: System.CommandLine 2.0.0-beta5+ migration guide; `VersionOption.Action` public property with `ClearsParseErrors = true` guidance.)

3. Bump `BrainyzCliVersion.Current` from `"0.3.0"` to `"0.4.0"` as part of the release commit.

**Output contract**: `brainz --version` prints exactly `brainyz <semver>\n` to stdout, exit code 0. The Homebrew `test do` (§4.1) and AUR `check()` (§4.4) assertions match this contract literally.

**Why now**: both lint steps above break on v0.4.0 without this fix.

**Test**: a new xUnit test in `Brainyz.Tests` that spawns the built CLI via `ProcessStartInfo`, runs with `--version`, and asserts both stdout equals `"brainyz 0.4.0\n"` and exit code is 0. The test reads the expected semver from `BrainyzCliVersion.Current` at compile time so bumping the constant does not require touching the test.

## 4. Per-channel manifests

### 4.1 Homebrew tap — `chaosfabric/homebrew-tap`

**Layout**:
```
Formula/
  brainyz.rb
README.md         ← one-liner pointing back at FavitoX/brainyz
```

**Formula skeleton** (generated by the updater job, illustrative):

```ruby
# Copyright 2026 Favio Andres Leyva
# SPDX-License-Identifier: Apache-2.0
class Brainyz < Formula
  desc "Global-first knowledge layer with per-project scoping and MCP integration"
  homepage "https://github.com/FavitoX/brainyz"
  version "0.4.0"
  license "Apache-2.0"

  on_macos do
    on_arm do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-osx-arm64.tar.gz"
      sha256 "<sha256-osx-arm64>"
    end
  end

  on_linux do
    on_intel do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-linux-x64.tar.gz"
      sha256 "<sha256-linux-x64>"
    end
    on_arm do
      url "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-linux-arm64.tar.gz"
      sha256 "<sha256-linux-arm64>"
    end
  end

  def install
    libexec.install "brainz"
    libexec.install Dir["*.so", "*.dylib"]
    bin.install_symlink libexec/"brainz"
    bin.install_symlink libexec/"brainz" => "bz"
  end

  test do
    assert_match "brainyz #{version}", shell_output("#{bin}/brainz --version")
  end
end
```

**Native-library handling**: `libsql.{so,dylib}` lives in `libexec/` next to the binary. macOS resolves sibling-dir dylibs via `@loader_path` (implicit for AOT single-file output). Linux resolves sibling `.so` files via rpath `$ORIGIN`, which NativeAOT sets by default on Linux publishes. The symlink-based install above preserves the libexec-relative resolution — there is no shim in between.

**Fallback if rpath/loader resolution breaks** (to be triggered only if verification §7.2 finds a failure on a real Homebrew install):

```ruby
def install
  libexec.install "brainz"
  libexec.install Dir["*.so", "*.dylib"]
  (bin/"brainz").write <<~SHIM
    #!/bin/bash
    if [[ "$(uname)" == "Darwin" ]]; then
      export DYLD_LIBRARY_PATH="#{libexec}:${DYLD_LIBRARY_PATH}"
    else
      export LD_LIBRARY_PATH="#{libexec}:${LD_LIBRARY_PATH}"
    fi
    exec "#{libexec}/brainz" "$@"
  SHIM
  chmod 0755, bin/"brainz"
  bin.install_symlink bin/"brainz" => "bz"
end
```

Verified by running `brew install` from a clean Homebrew in a CI matrix (see §7.1).

**`bz` alias**: installed as a `bin.install_symlink` pointing at the `libexec/brainz` binary (same install path whether the shim-free or shim-with-LD_LIBRARY_PATH variant is used).

**macOS Intel (`osx-x64`) is not published** by the current `release.yml` (the matrix dropped it per the comment "bring osx-x64 back if there's demand"). The formula consequently has no `on_intel` block inside `on_macos`. Homebrew surfaces a clean "formula not available for your platform" error if an Intel-Mac user tries to install — acceptable for v0.4. If demand appears, adding `osx-x64` is a release-matrix edit + one new `on_intel do ... end` block in the formula, not a redesign.

**linux-arm64 note**: the archive ships without `libsql.so` (Nelknet.LibSQL.Bindings 0.2.5 does not bundle one — documented in `nelknet-bugs.md` §8). The Homebrew formula ships the binary anyway; `brew install` succeeds and `brainz --version` works. Any command that opens the DB fails with a clear message directing users to supply their own aarch64 `libsql.so`. This matches the non-blocking smoke-test policy already in `release.yml`.

### 4.2 Scoop bucket — `chaosfabric/scoop-bucket`

**Layout**:
```
bucket/
  brainyz.json
README.md
```

**Manifest skeleton**:

```json
{
  "version": "0.4.0",
  "description": "Global-first knowledge layer with per-project scoping and MCP integration",
  "homepage": "https://github.com/FavitoX/brainyz",
  "license": "Apache-2.0",
  "architecture": {
    "64bit": {
      "url": "https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-win-x64.zip",
      "hash": "<sha256-win-x64>"
    }
  },
  "bin": [
    "brainz.exe",
    ["brainz.exe", "bz"]
  ],
  "checkver": "github",
  "autoupdate": {
    "architecture": {
      "64bit": {
        "url": "https://github.com/FavitoX/brainyz/releases/download/v$version/brainz-win-x64.zip"
      }
    }
  }
}
```

**How the `bin` array works**: Scoop reads each entry independently:
- The bare string `"brainz.exe"` creates a shim named `brainz` on `PATH` pointing at the bundled `brainz.exe`.
- The nested array `["brainz.exe", "bz"]` creates a separate shim named `bz` that also invokes `brainz.exe`.

Both entries together produce two shims (`brainz`, `bz`); the first alone would only produce `brainz`. `libsql.dll` lives alongside `brainz.exe` in the extracted directory; the Windows loader finds it next to the executable with no extra config.

`checkver` + `autoupdate` are defensive: our updater job pushes the manifest directly, so autoupdate is not needed in the normal path — but leaving it in means Scoop's own nightly check keeps the bucket in sync if our CI ever stops pushing.

### 4.3 Winget — `chaosfabric/winget-manifests` (shadow) + PR to `microsoft/winget-pkgs`

**Shadow-repo layout** (mirrors the path microsoft/winget-pkgs expects):
```
manifests/c/ChaosFabric/Brainyz/
  0.4.0/
    ChaosFabric.Brainyz.yaml
    ChaosFabric.Brainyz.installer.yaml
    ChaosFabric.Brainyz.locale.en-US.yaml
```

**Manifests generated by `wingetcreate new --urls <url>.zip` locally, then hand-polished**. Key fields:

- `PackageIdentifier: ChaosFabric.Brainyz`
- `Publisher: ChaosFabric`
- `PackageName: Brainyz`
- `License: Apache-2.0`
- `PublisherUrl: https://github.com/chaosfabric`
- `PackageUrl: https://github.com/FavitoX/brainyz`
- `Installers:` single entry with `Architecture: x64`, `InstallerType: zip`, the release URL and SHA256, plus two `NestedInstallerFiles` entries sharing the same `RelativeFilePath` but exposing two distinct `PortableCommandAlias` values (this is how winget registers PATH aliases for `portable` installs — the top-level `Commands:` field is metadata-only and does **not** affect PATH):

  ```yaml
  Installers:
    - Architecture: x64
      InstallerType: zip
      NestedInstallerType: portable
      InstallerUrl: https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-win-x64.zip
      InstallerSha256: <sha256-win-x64>
      NestedInstallerFiles:
        - RelativeFilePath: brainz.exe
          PortableCommandAlias: brainz
        - RelativeFilePath: brainz.exe
          PortableCommandAlias: bz
  ```

  (Per winget-pkgs schema 1.6+, `PortableCommandAlias` is the only mechanism for creating PATH entries for a portable install. Two entries against the same `RelativeFilePath` produce two aliases.)

- `Commands: [brainz, bz]` may be included in the default-locale file as informational metadata (`winget search` surfaces it), but it does NOT create shims or PATH entries.

**Submission flow on v0.4.0**:
1. `wingetcreate new https://github.com/FavitoX/brainyz/releases/download/v0.4.0/brainz-win-x64.zip` on a Windows box.
2. Answer the prompts; accept the generated SHA256.
3. Commit the three YAMLs to `chaosfabric/winget-manifests`.
4. `wingetcreate submit --token <PAT>` opens the PR to `microsoft/winget-pkgs`.
5. Wait 2–7 days for review + smoke test. Address any lint findings.
6. Once merged, `winget install ChaosFabric.Brainyz` works for everyone.

**Automation deferred to v0.5+**: a CI job that runs `wingetcreate update --urls <v0.5.0-url> --version 0.5.0 ChaosFabric.Brainyz --submit --token $WINGET_PAT` on each stable tag. Prereq is a GitHub PAT with `public_repo` scope stored in secrets; that's a cost we take after the first manual PR proves the manifests.

### 4.4 AUR — `chaosfabric/aur-brainyz-bin` (shadow) + direct push to AUR

**Shadow-repo layout**:
```
PKGBUILD
.SRCINFO
README.md
```

**`PKGBUILD` skeleton**:

```bash
# Maintainer: Favio Leyva <andy.carp@gmail.com>
pkgname=brainyz-bin
_pkgname=brainyz
pkgver=0.4.0
pkgrel=1
pkgdesc="Global-first knowledge layer with per-project scoping and MCP integration"
arch=('x86_64')
url="https://github.com/FavitoX/brainyz"
license=('Apache-2.0')
provides=('brainyz')
conflicts=('brainyz')
source_x86_64=("$pkgname-$pkgver.tar.gz::https://github.com/FavitoX/brainyz/releases/download/v$pkgver/brainz-linux-x64.tar.gz")
sha256sums_x86_64=('<sha256-linux-x64>')

package() {
  # Install binary + sibling libsql.so into /usr/lib/brainyz/; NativeAOT's
  # $ORIGIN rpath resolves the .so relative to the binary's own directory.
  install -Dm755 "$srcdir/brainz"     "$pkgdir/usr/lib/brainyz/brainz"
  install -Dm755 "$srcdir/libsql.so"  "$pkgdir/usr/lib/brainyz/libsql.so"
  # /usr/bin shims (symlinks, not wrappers — $ORIGIN resolves through symlinks).
  install -d "$pkgdir/usr/bin"
  ln -s ../lib/brainyz/brainz "$pkgdir/usr/bin/brainz"
  ln -s ../lib/brainyz/brainz "$pkgdir/usr/bin/bz"
}

check() {
  "$srcdir/brainz" --version | grep -Fq "brainyz $pkgver"
}
```

**Native-library resolution**: NativeAOT on Linux sets the binary's rpath to `$ORIGIN`. Per `ld.so(8)` (`man7.org/linux/man-pages/man8/ld.so.8.html`) and glibc's `elf/dl-load.c`, `$ORIGIN` is resolved using `readlink("/proc/self/exe")`, which transparently follows symlinks. So the `/usr/bin/brainz → /usr/lib/brainyz/brainz` symlink gives `$ORIGIN = /usr/lib/brainyz`, and `libsql.so` sitting in the same directory is found. This is verified by running `LD_DEBUG=libs /usr/bin/brainz --version 2>&1 | grep libsql` during the bootstrap (listed in the release checklist in §7.2).

**Fallback if rpath resolution breaks**: if `LD_DEBUG=libs` shows libsql.so not found via `$ORIGIN`, replace the symlink shims with explicit wrappers that set `LD_LIBRARY_PATH`:

```bash
install -d "$pkgdir/usr/bin"
for name in brainz bz; do
  cat > "$pkgdir/usr/bin/$name" <<'EOF'
#!/bin/sh
export LD_LIBRARY_PATH="/usr/lib/brainyz:${LD_LIBRARY_PATH}"
exec "/usr/lib/brainyz/brainz" "$@"
EOF
  chmod 0755 "$pkgdir/usr/bin/$name"
done
```

This fallback path stays out of the primary PKGBUILD unless verified necessary.

**`options=(!strip)`**: omitted intentionally. NativeAOT publishes on Linux do NOT strip by default unless `<StripSymbols>true</StripSymbols>` is set in the csproj (it is not). Using `!strip` would disable makepkg's default stripping and ship a larger package for no reason. Letting `makepkg` strip is safe — AOT binaries don't carry JIT metadata that stripping would break.

**`license=('Apache-2.0')`**: accepted by `pacman` ≥ 7.0 (released 2024). Arch rolling-release users are on ≥ 7.0 as of 2026; Arch derivatives on older pacman would fall back to `license=('Apache')`. Verified-at-bootstrap item (§11).

**`.SRCINFO` is generated with `makepkg --printsrcinfo > .SRCINFO`**; AUR tooling requires it to be committed alongside the PKGBUILD.

**Submission flow on v0.4.0**:
1. On Arch (WSL / VM / container): register an AUR account, add SSH public key in the web profile.
2. `git clone ssh://aur@aur.archlinux.org/brainyz-bin.git` (first push creates the repo — it starts empty).
3. Copy the PKGBUILD + .SRCINFO in, test locally with `makepkg -si`.
4. `git add PKGBUILD .SRCINFO && git commit -m "v0.4.0" && git push`.
5. Snapshot the same files in `chaosfabric/aur-brainyz-bin` for GitHub-side review/history.

**Automation deferred to v0.5+**: a CI job that takes `pkgver` from the tag, templates PKGBUILD + `.SRCINFO`, then `git push` to AUR over SSH with a deploy key in secrets.

**ARM64 not covered**: `arch=('x86_64')` only. `aarch64` can be added later when Nelknet ships the native library for it (nelknet-bugs.md §8).

## 5. Release-workflow changes

### 5.1 Job topology

Before:
```
build ─► release
```

After:
```
build ─► checksums ─► release ─► update-homebrew-tap (parallel)
                               └► update-scoop-bucket (parallel)
```

### 5.2 `update-homebrew-tap` (new)

```yaml
update-homebrew-tap:
  name: update Homebrew tap
  needs: release
  runs-on: ubuntu-latest
  # Stable tags only. Guard against both tag-push and workflow_dispatch:
  # on tag-push, ref_name is the tag; on workflow_dispatch, ref_name is
  # the branch, so we OR-check the input tag too.
  if: >
    !contains(github.event.inputs.tag || github.ref_name, '-')
  continue-on-error: true
  steps:
    - name: Compute release metadata
      id: meta
      run: |
        TAG="${{ github.event.inputs.tag || github.ref_name }}"
        VERSION="${TAG#v}"
        echo "tag=$TAG"         >> "$GITHUB_OUTPUT"
        echo "version=$VERSION" >> "$GITHUB_OUTPUT"
    - name: Fetch SHA256SUMS
      run: |
        curl -fsSL -o SHA256SUMS.txt \
          "https://github.com/FavitoX/brainyz/releases/download/${{ steps.meta.outputs.tag }}/SHA256SUMS.txt"
    - name: Render Formula
      run: ./tools/render-homebrew-formula.sh \
             ${{ steps.meta.outputs.version }} \
             ${{ steps.meta.outputs.tag }} \
             SHA256SUMS.txt \
             > Formula-brainyz.rb
    - name: Push to chaosfabric/homebrew-tap
      env:
        DEPLOY_KEY: ${{ secrets.HOMEBREW_TAP_DEPLOY_KEY }}
      run: |
        mkdir -p ~/.ssh && echo "$DEPLOY_KEY" > ~/.ssh/id_ed25519
        chmod 0600 ~/.ssh/id_ed25519
        ssh-keyscan github.com >> ~/.ssh/known_hosts
        git clone git@github.com:chaosfabric/homebrew-tap.git tap
        cp Formula-brainyz.rb tap/Formula/brainyz.rb
        cd tap
        git -c user.name='brainyz-release-bot' \
            -c user.email='41898282+github-actions[bot]@users.noreply.github.com' \
            commit -am "brainyz ${{ steps.meta.outputs.version }}"
        git push
```

(The commit author email uses the GitHub-canonical `noreply` pattern instead of a `@chaosfabric.dev` address that may or may not be owned yet — the domain question is in §11 open items.)

### 5.3 `update-scoop-bucket` (new)

Mirror of 5.2: renders `bucket/brainyz.json` from a second template script `tools/render-scoop-manifest.sh`, pushes to `chaosfabric/scoop-bucket` via deploy key (`SCOOP_BUCKET_DEPLOY_KEY`).

### 5.4 Template scripts

Live under `tools/`:
- `tools/render-homebrew-formula.sh` — bash, renders `tools/templates/homebrew-brainyz.rb.tmpl`.
- `tools/render-scoop-manifest.sh` — bash, renders `tools/templates/scoop-brainyz.json.tmpl`.

**Contract** (both scripts):
- **Arguments**: `$1 = version` (e.g. `0.4.0`), `$2 = tag` (e.g. `v0.4.0`), `$3 = path to SHA256SUMS.txt`.
- **Stdin**: none.
- **Stdout**: rendered manifest.
- **Stderr**: human-readable errors on bad input.
- **Exit**: `0` on success, `1` on any missing / malformed input.

**Parse logic**: extract per-RID SHA256s from the sums file using a single `awk` pass:
```bash
sha_for() {
  awk -v f="$1" '$2 == f { print $1; found=1 } END { exit !found }' "$3"
}
SHA_OSX_ARM64=$(sha_for  "brainz-osx-arm64.tar.gz")  || { echo "missing osx-arm64 checksum" >&2; exit 1; }
SHA_LINUX_X64=$(sha_for  "brainz-linux-x64.tar.gz")  || { echo "missing linux-x64 checksum" >&2; exit 1; }
SHA_LINUX_ARM64=$(sha_for "brainz-linux-arm64.tar.gz") || { echo "missing linux-arm64 checksum" >&2; exit 1; }
SHA_WIN_X64=$(sha_for    "brainz-win-x64.zip")       || { echo "missing win-x64 checksum" >&2; exit 1; }
```

Then `sed` on the template:
```bash
sed -e "s/{{VERSION}}/$1/g" \
    -e "s/{{TAG}}/$2/g" \
    -e "s/{{SHA256_OSX_ARM64}}/$SHA_OSX_ARM64/" \
    -e "s/{{SHA256_LINUX_X64}}/$SHA_LINUX_X64/" \
    -e "s/{{SHA256_LINUX_ARM64}}/$SHA_LINUX_ARM64/" \
    tools/templates/homebrew-brainyz.rb.tmpl
```

Missing checksum lines are a hard error (non-zero exit), never silent empty substitutions — a stale or broken `SHA256SUMS.txt` must never produce a formula with empty hash fields.

**Unit testable** with a golden `SHA256SUMS.txt` fixture (`tests/fixtures/SHA256SUMS.example.txt`) and a recorded expected output. Test script lives at `tests/template-render.test.sh`; called from the new `packaging.yml` on every run (§7.1).

### 5.5 Secrets required (one-time setup)

- `HOMEBREW_TAP_DEPLOY_KEY` — SSH private key with write access to `chaosfabric/homebrew-tap` (ed25519 recommended).
- `SCOOP_BUCKET_DEPLOY_KEY` — same for `chaosfabric/scoop-bucket`.

Deploy keys (vs. PAT) scope access to exactly those two repos; compromise of the key does not grant access to anything else in the org.

## 7. Testing and verification

### 7.1 Automated in CI (new)

- **`SHA256SUMS.txt` validation** — the `release` job runs `sha256sum -c SHA256SUMS.txt` before uploading to the GitHub Release, catching corrupted artifacts locally before publish. Always on `ubuntu-latest` runner (Linux `sha256sum`; macOS/Windows would need `shasum -a 256` / `certutil -hashfile`).

- **Template-render smoke** — unit-level tests that both render scripts produce valid manifests given a golden `SHA256SUMS.txt` fixture:
  - `tools/render-homebrew-formula.sh` → pipe through `ruby -c` (syntax check).
  - `tools/render-scoop-manifest.sh` → pipe through `jq empty` (JSON validity).
  - Each run diffed against a recorded golden output (`tests/fixtures/homebrew-brainyz.rb.expected`, `tests/fixtures/scoop-brainyz.json.expected`) so unintended template changes are caught.

- **`packaging.yml` — weekly verification workflow** (new). Two concerns folded into one workflow because they share context:

  ```yaml
  name: Packaging verification
  on:
    schedule: [{cron: '17 6 * * 1'}]   # Mondays 06:17 UTC
    workflow_dispatch: {}

  jobs:
    template-smoke:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v6
        - run: bash tests/template-render.test.sh

    homebrew-audit:
      runs-on: macos-latest
      steps:
        - run: |
            brew tap chaosfabric/tap
            brew audit --strict --online chaosfabric/tap/brainyz

    drift-detection:
      runs-on: ubuntu-latest
      steps:
        - name: Latest GitHub Release
          id: gh
          run: |
            LATEST=$(curl -fsSL https://api.github.com/repos/FavitoX/brainyz/releases/latest | jq -r .tag_name)
            echo "version=${LATEST#v}" >> "$GITHUB_OUTPUT"

        - name: Homebrew tap version
          id: brew
          run: |
            VER=$(curl -fsSL https://raw.githubusercontent.com/chaosfabric/homebrew-tap/HEAD/Formula/brainyz.rb \
                  | grep -oE 'version "[^"]+"' | head -n1 | cut -d'"' -f2)
            echo "version=$VER" >> "$GITHUB_OUTPUT"

        - name: Scoop bucket version
          id: scoop
          run: |
            VER=$(curl -fsSL https://raw.githubusercontent.com/chaosfabric/scoop-bucket/HEAD/bucket/brainyz.json \
                  | jq -r .version)
            echo "version=$VER" >> "$GITHUB_OUTPUT"

        - name: Winget manifest version
          id: winget
          env:
            GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          run: |
            # Manifest folder path IS the version; list and take the latest by natural sort.
            VER=$(gh api repos/microsoft/winget-pkgs/contents/manifests/c/ChaosFabric/Brainyz \
                  --jq '.[].name' | sort -V | tail -n1)
            echo "version=$VER" >> "$GITHUB_OUTPUT"

        - name: AUR version
          id: aur
          run: |
            # RPC returns "Version":"0.4.0-1" — strip the -<pkgrel> suffix.
            VER=$(curl -fsSL 'https://aur.archlinux.org/rpc/?v=5&type=info&arg=brainyz-bin' \
                  | jq -r '.results[0].Version // empty' \
                  | sed 's/-[0-9]*$//')
            echo "version=$VER" >> "$GITHUB_OUTPUT"

        - name: Compare and open/update drift issue
          uses: actions/github-script@v8
          env:
            GH_VER:     ${{ steps.gh.outputs.version }}
            BREW_VER:   ${{ steps.brew.outputs.version }}
            SCOOP_VER:  ${{ steps.scoop.outputs.version }}
            WINGET_VER: ${{ steps.winget.outputs.version }}
            AUR_VER:    ${{ steps.aur.outputs.version }}
          with:
            script: |
              const latest = process.env.GH_VER;
              const channels = {
                Homebrew: process.env.BREW_VER,
                Scoop:    process.env.SCOOP_VER,
                Winget:   process.env.WINGET_VER,
                AUR:      process.env.AUR_VER,
              };
              const stale = Object.entries(channels).filter(([_, v]) => v !== latest);
              if (stale.length === 0) return;   // in sync, nothing to do

              const title = `Packaging drift: latest release is ${latest}`;
              const body  = [
                `GitHub Release: **${latest}**`,
                '',
                '| Channel | Version | Status |',
                '|---|---|---|',
                ...Object.entries(channels).map(([c, v]) =>
                  `| ${c} | ${v || '(missing)'} | ${v === latest ? 'OK' : 'STALE'} |`),
                '',
                'Detected by `.github/workflows/packaging.yml`.'
              ].join('\n');

              // Idempotent: find an existing open issue with the same title prefix.
              const {data: issues} = await github.rest.issues.listForRepo({
                owner: context.repo.owner, repo: context.repo.repo,
                state: 'open', labels: 'packaging-drift', per_page: 100,
              });
              const existing = issues.find(i => i.title === title);

              if (existing) {
                await github.rest.issues.update({
                  owner: context.repo.owner, repo: context.repo.repo,
                  issue_number: existing.number, body,
                });
              } else {
                await github.rest.issues.create({
                  owner: context.repo.owner, repo: context.repo.repo,
                  title, body, labels: ['packaging-drift'],
                });
              }
  ```

  - **Trigger**: weekly + manual dispatch. Not release-gated — this job exists specifically to catch the cases where the release *didn't* propagate.
  - **Runner choice**: `macos-latest` for the `brew audit` job (Homebrew's `brew` CLI is macOS-first; it works on Linux but arm64 runners are slower for installs); `ubuntu-latest` for the rest.
  - **Failure handling**: drift triggers a GitHub issue (idempotent — reuses an existing open issue for the same drift target); `brew audit` failure is surfaced as a workflow failure and also opens an issue.
  - **Public repos only** — Homebrew tap and Scoop bucket are public, no auth needed. AUR RPC and winget-pkgs raw URLs are also public.

### 7.2 Manual verification per release

Checklist embedded in `docs/packaging.md`, run once per stable release against a clean machine. Structured per-channel (not per-platform) so channel-specific alias bugs surface clearly:

- [ ] **Homebrew (macOS arm64)**: `brew install chaosfabric/tap/brainyz` → `brainz --version` prints `brainyz <ver>` → `bz --version` prints the same.
- [ ] **Homebrew (Linux x64)**: same, in a fresh Ubuntu container (`brew` on Linux is supported; the formula is platform-agnostic).
- [ ] **Scoop (Windows)**: `scoop install chaosfabric/brainyz` → `brainz --version` → `bz --version`.
- [ ] **Winget (Windows)**: `winget install ChaosFabric.Brainyz` → `brainz --version` → `bz --version`.
- [ ] **AUR (Arch Linux)**: `yay -S brainyz-bin` → `brainz --version` → `bz --version`; additionally run `LD_DEBUG=libs brainz --version 2>&1 | grep libsql` to confirm `libsql.so` resolves via `$ORIGIN` rpath (not via `LD_LIBRARY_PATH`).
- [ ] Each channel: after `--version` check, run one full smoke cycle — `brainz init --here` in a temp dir, `brainz add note "smoke"`, `brainz list`, confirm non-empty output. Catches runtime failures that `--version` alone misses.

### 7.3 Smoke-test binary (unchanged)

`tools/smoke-test.cs` keeps running per-RID against the raw published binary. Packaging layers sit on top of that; if the binary is broken, packaging never runs.

### 7.4 Rollback procedure

Per channel, documented in `docs/packaging.md`:

| Channel | Rollback action |
|---|---|
| Homebrew tap | `git revert <bad-commit> && git push` on `chaosfabric/homebrew-tap`. |
| Scoop bucket | `git revert <bad-commit> && git push` on `chaosfabric/scoop-bucket`. |
| Winget | Open a PR to `microsoft/winget-pkgs` removing the offending version's manifest folder. |
| AUR | `git push` the previous PKGBUILD to `ssh://aur@aur.archlinux.org/brainyz-bin.git`. |

The GitHub Release itself is never edited — the underlying archives remain available for manual install at all times. A user who already installed the bad version downgrades with their package manager's standard mechanism.

## 8. Error handling and failure modes

### 8.1 Updater-job failures

Both `update-homebrew-tap` and `update-scoop-bucket` are `continue-on-error: true`. Failure modes and recovery:

- **Deploy key rejected** → check the secret is populated; re-run the job on the same tag (idempotent).
- **SHA256SUMS.txt not yet visible** → GitHub Release propagation delay (rare, <1 min). Re-run the job.
- **Template script produces invalid file** → caught locally by the §7.1 smoke test; if it lands in CI it's an advisory failure, tap/bucket is untouched, fix the template and re-run.

Every failure leaves prior state intact — formula/manifest is not half-written.

### 8.2 Manual-submission failures

- **Winget PR rejected** (validation errors from Microsoft's CI) → fix the manifest in `chaosfabric/winget-manifests`, push a new commit to the PR.
- **Winget SmartScreen warnings** — unsigned binary. Expected per §1 "out of scope"; `docs/packaging.md` includes a note for users that "More info → Run anyway" is the current path until signing lands.
- **AUR build fails** on a user's machine → `makepkg -si` error lines; bug filed against `chaosfabric/aur-brainyz-bin` (the shadow repo) for visibility, fix pushed to AUR directly.

### 8.3 Package manager outages

Each channel failing in isolation is tolerable — users have four paths to install, plus the raw tar.gz/zip. No error-code system needed; these failures surface as HTTP errors or package-manager messages, outside the app boundary.

## 9. Security considerations

- **Checksums are the trust root.** Every channel pins to the SHA256 published in the GitHub Release. An attacker who compromised the tap/bucket repos could only change which release is installed (all real binaries are in FavitoX/brainyz/releases, which has its own auth and audit trail).
- **Deploy keys are single-repo scoped.** Compromise of the Homebrew deploy key does not grant access to the Scoop bucket or the main repo.
- **No secrets flow through manifests.** The manifests are pure metadata + URLs + SHA256; nothing a package-manager user downloads is derived from our secrets.
- **Unsigned binaries** — Microsoft SmartScreen shows "Windows protected your PC" on first run of the `brainz.exe` installed by Winget; user clicks "More info → Run anyway." macOS Gatekeeper applies `com.apple.quarantine` to manually-downloaded archives, but Homebrew's own download pipeline is not quarantined — so `brew install` users do NOT hit Gatekeeper, only users who `curl` the tar.gz directly and then run the binary. Both documented in `docs/packaging.md`. Code signing is deferred per §1.

## 10. Consequences and follow-ups

### 10.1 Ongoing cost

- Homebrew tap + Scoop bucket: zero manual work per release (CI handles it).
- Winget: ~10 min per release to run `wingetcreate update` + submit, until the automation lands in v0.5+.
- AUR: ~5 min per release to push a new `pkgver`, until automation lands.

Total human-in-the-loop per stable release in v0.4: ~15 minutes; ~0 minutes from v0.5+ once Winget/AUR automation lands.

### 10.2 Observability

A one-liner weekly cron that curls `https://api.github.com/repos/FavitoX/brainyz/releases/latest` vs. each package manager's latest and reports drift. Nice-to-have, tracked in `docs/packaging.md` futures section. Out of scope for v0.4.

### 10.3 When to revisit

- **Code signing** — when SmartScreen friction appears in user reports.
- **nix flake** — when an issue requests it.
- **Homebrew core submission** — after six consecutive stable releases with no regressions, AND the project meets Homebrew's own acceptability criteria (notable, actively maintained, stable versioning, not a fork, passes `brew audit --new --strict`). Reference: Homebrew's `Acceptable Formulae` doc.
- **AUR source package** — if a contributor opens a PR (we won't chase it ourselves).
