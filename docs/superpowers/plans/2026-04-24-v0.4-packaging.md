# v0.4 Packaging extendido — Implementation Plan

> **For agentic workers:** REQUIRED: Use `superpowers:subagent-driven-development` (if subagents available) or `superpowers:executing-plans` to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Publish brainyz through four native package managers (Homebrew tap, Scoop bucket, Winget, AUR-bin) by extending the release pipeline with checksum generation and two auto-push updater jobs, plus manual-bootstrap templates for the Winget and AUR channels — per the design spec at [docs/superpowers/specs/2026-04-24-v0.4-packaging-design.md](../specs/2026-04-24-v0.4-packaging-design.md).

**Architecture:** One small app-code change (wire `brainz --version` to `BrainyzCliVersion.Current` via a `VersionOption.Action` override — precondition for the package-manager lint/test assertions). The rest is CI plumbing: a new `checksums` job producing `SHA256SUMS.txt`, two updater jobs (`update-homebrew-tap`, `update-scoop-bucket`) that render template manifests and push to the `chaosfabric/*` repos via deploy keys, shadow repos (`chaosfabric/winget-manifests`, `chaosfabric/aur-brainyz-bin`) populated by hand on v0.4.0, and a separate weekly `packaging.yml` workflow that detects drift and opens/updates a GitHub issue.

**Tech Stack:** .NET 10, `System.CommandLine` 2.0.7 (custom `SynchronousCommandLineAction` to override `VersionOption`), xUnit, bash + awk + sed for render scripts, GitHub Actions (`actions/checkout@v6`, `actions/download-artifact@v8`, `softprops/action-gh-release@v3`, `actions/github-script@v8`), SSH deploy keys (ed25519) for tap/bucket pushes, `wingetcreate` + AUR SSH for the manual bootstrap.

### Task 1.2: `PrintVersionAction`

Write `src/Brainyz.Cli/PrintVersionAction.cs`:

```csharp
// Copyright 2026 Favio Andres Leyva
// SPDX-License-Identifier: Apache-2.0

using System.CommandLine;
using System.CommandLine.Invocation;

namespace Brainyz.Cli;

/// <summary>
/// Custom action for the built-in <c>--version</c> option. Overrides the
/// default <see cref="VersionOption"/> behavior (which prints the assembly
/// version) so the CLI emits <c>"brainyz &lt;semver&gt;"</c> sourced from
/// <see cref="BrainyzCliVersion.Current"/>. Required so every package
/// manager's post-install lint or test step can assert on a stable format.
/// </summary>
internal sealed class PrintVersionAction : SynchronousCommandLineAction
{
    public PrintVersionAction()
    {
        // Without this, a parse error on an unrelated option would block the
        // --version output. Matches the default VersionOption.Action behavior.
        Terminating = true;
    }

    public override int Invoke(ParseResult parseResult)
    {
        Console.WriteLine($"brainyz {BrainyzCliVersion.Current}");
        return 0;
    }
}
```

**Note on `ClearsParseErrors`**: the spec referenced `ClearsParseErrors = true` but in System.CommandLine 2.0.0-beta5+ the equivalent is the `Terminating = true` property on `CommandLineAction`. If your local SDK reports `ClearsParseErrors` as the real property name, use that instead — both are advisory to the parser that this action runs even if other parse errors occurred. The smoke test in Task 1.3 catches any mis-wiring.

### Task 1.4: xUnit subprocess test for `--version` output

**Files:**
- Create: `tests/Brainyz.Tests/VersionOutputTests.cs`

- [ ] **Step 1: Write the failing test**

Create `tests/Brainyz.Tests/VersionOutputTests.cs`:

```csharp
// Copyright 2026 Favio Andres Leyva
// SPDX-License-Identifier: Apache-2.0

using System.Diagnostics;
using Brainyz.Cli;

namespace Brainyz.Tests;

/// <summary>
/// Contract test: <c>brainz --version</c> must print exactly
/// <c>brainyz &lt;semver&gt;</c> (plus a trailing newline) to stdout and exit 0.
/// Package managers' post-install lint/test hooks assert on this literal shape,
/// so drift breaks every channel's CI at once.
/// </summary>
public class VersionOutputTests
{
    [Fact]
    public async Task Version_flag_prints_canonical_format()
    {
        string exePath = ResolveBrainzBinary();

        var psi = new ProcessStartInfo(exePath, "--version")
        {
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false,
        };
        using var proc = Process.Start(psi)
            ?? throw new InvalidOperationException("failed to spawn brainz");

        string stdout = await proc.StandardOutput.ReadToEndAsync();
        string stderr = await proc.StandardError.ReadToEndAsync();
        await proc.WaitForExitAsync();

        Assert.Equal(0, proc.ExitCode);
        Assert.Equal(string.Empty, stderr);
        Assert.Equal($"brainyz {BrainyzCliVersion.Current}{Environment.NewLine}", stdout);
    }

    /// <summary>
    /// Locate the built CLI binary. Tests run under
    /// <c>tests/Brainyz.Tests/bin/&lt;config&gt;/net10.0/</c>; the CLI build sits at
    /// <c>src/Brainyz.Cli/bin/&lt;config&gt;/net10.0/brainz[.exe]</c>.
    /// </summary>
    private static string ResolveBrainzBinary()
    {
        string testDir = AppContext.BaseDirectory;
        // bin/Release/net10.0/ → repo root is four levels up (tests/Brainyz.Tests/).
        string repoRoot = Path.GetFullPath(Path.Combine(testDir, "..", "..", "..", "..", ".."));
        string tfm = Path.GetFileName(Path.GetDirectoryName(testDir.TrimEnd(Path.DirectorySeparatorChar)))!;
        string configDir = Path.GetFileName(Path.GetDirectoryName(Path.GetDirectoryName(testDir.TrimEnd(Path.DirectorySeparatorChar))))!;

        string fileName = OperatingSystem.IsWindows() ? "brainz.exe" : "brainz";
        string cliBinary = Path.Combine(repoRoot, "src", "Brainyz.Cli", "bin", configDir, tfm, fileName);
        if (!File.Exists(cliBinary))
        {
            throw new FileNotFoundException(
                $"Built CLI binary not found at {cliBinary}. " +
                "Run `dotnet build src/Brainyz.Cli/Brainyz.Cli.csproj` before running this test.",
                cliBinary);
        }
        return cliBinary;
    }
}
```

- [ ] **Step 2: Insert the `checksums` job between `build` and `release`**

After the `build` job's closing indentation but before `release:`, add:

```yaml
  checksums:
    name: generate SHA256SUMS
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v8
        with:
          path: artifacts
          merge-multiple: true

      - name: Compute SHA256SUMS
        run: |
          cd artifacts
          # Skip any artifact that wasn't produced (e.g., linux-arm64 smoke
          # skip does not remove the archive; this list is literal).
          sha256sum brainz-linux-x64.tar.gz \
                    brainz-linux-arm64.tar.gz \
                    brainz-osx-arm64.tar.gz \
                    brainz-win-x64.zip \
            > SHA256SUMS.txt
          cat SHA256SUMS.txt

      - name: Verify checksums
        run: |
          cd artifacts
          sha256sum -c SHA256SUMS.txt

      - uses: actions/upload-artifact@v7
        with:
          name: checksums
          path: artifacts/SHA256SUMS.txt
```

Then modify the existing `release` job:

- Change `needs: build` to `needs: [build, checksums]`.
- In the `Create Release` step, append `SHA256SUMS.txt` to the `files:` glob. Current shape:
  ```yaml
          files: artifacts/brainz-*.*
  ```
  becomes:
  ```yaml
          files: |
            artifacts/brainz-*.*
            artifacts/SHA256SUMS.txt
  ```

### Task 3.3: `render-homebrew-formula.sh`

**Files:**
- Create: `tools/render-homebrew-formula.sh`

- [ ] **Step 1: Write the script**

Create `tools/render-homebrew-formula.sh`:

```bash
#!/usr/bin/env bash
# Render tools/templates/homebrew-brainyz.rb.tmpl for a given release.
#
# Usage: render-homebrew-formula.sh <version> <tag> <sha256sums-path>
#   version          — semver without 'v', e.g. "0.4.0"
#   tag              — tag name with 'v', e.g. "v0.4.0"
#   sha256sums-path  — path to a SHA256SUMS.txt as emitted by `sha256sum`
#
# Stdout: rendered Ruby formula.
# Stderr: human-readable errors on any missing/malformed input.
# Exit:   0 on success, 1 on any error.

set -euo pipefail

if [[ $# -ne 3 ]]; then
  echo "usage: $0 <version> <tag> <sha256sums-path>" >&2
  exit 1
fi

VERSION="$1"
TAG="$2"
SUMS="$3"

if [[ ! -f "$SUMS" ]]; then
  echo "error: SHA256SUMS file not found: $SUMS" >&2
  exit 1
fi

sha_for() {
  local file="$1"
  local sha
  sha=$(awk -v f="$file" '$2 == f { print $1; found=1 } END { exit !found }' "$SUMS") || {
    echo "error: missing checksum for $file in $SUMS" >&2
    return 1
  }
  printf '%s' "$sha"
}

SHA_OSX_ARM64=$(sha_for   "brainz-osx-arm64.tar.gz")
SHA_LINUX_X64=$(sha_for   "brainz-linux-x64.tar.gz")
SHA_LINUX_ARM64=$(sha_for "brainz-linux-arm64.tar.gz")

TMPL_DIR="$(cd "$(dirname "$0")/templates" && pwd)"
sed -e "s/{{VERSION}}/$VERSION/g" \
    -e "s/{{TAG}}/$TAG/g" \
    -e "s/{{SHA256_OSX_ARM64}}/$SHA_OSX_ARM64/g" \
    -e "s/{{SHA256_LINUX_X64}}/$SHA_LINUX_X64/g" \
    -e "s/{{SHA256_LINUX_ARM64}}/$SHA_LINUX_ARM64/g" \
    "$TMPL_DIR/homebrew-brainyz.rb.tmpl"
```

- Modify: `.github/workflows/release.yml`

After the `release` job (at the end of `jobs:`), add:

```yaml
  update-homebrew-tap:
    name: update Homebrew tap
    needs: release
    runs-on: ubuntu-latest
    # Stable tags only. Guard handles both tag-push and workflow_dispatch:
    # on tag-push ref_name is the tag; on workflow_dispatch ref_name is the
    # branch, so we OR-check the input tag as well.
    if: >
      !contains(github.event.inputs.tag || github.ref_name, '-')
    continue-on-error: true
    steps:
      - uses: actions/checkout@v6

      - name: Compute release metadata
        id: meta
        run: |
          TAG="${{ github.event.inputs.tag || github.ref_name }}"
          VERSION="${TAG#v}"
          echo "tag=$TAG"         >> "$GITHUB_OUTPUT"
          echo "version=$VERSION" >> "$GITHUB_OUTPUT"

      - name: Fetch SHA256SUMS from release
        run: |
          curl -fsSL -o SHA256SUMS.txt \
            "https://github.com/FavitoX/brainyz/releases/download/${{ steps.meta.outputs.tag }}/SHA256SUMS.txt"

      - name: Render Formula
        run: |
          bash tools/render-homebrew-formula.sh \
            "${{ steps.meta.outputs.version }}" \
            "${{ steps.meta.outputs.tag }}" \
            SHA256SUMS.txt \
            > Formula-brainyz.rb
          head -30 Formula-brainyz.rb

      - name: Push to chaosfabric/homebrew-tap
        env:
          DEPLOY_KEY: ${{ secrets.HOMEBREW_TAP_DEPLOY_KEY }}
        run: |
          mkdir -p ~/.ssh
          printf '%s' "$DEPLOY_KEY" > ~/.ssh/id_ed25519
          chmod 0600 ~/.ssh/id_ed25519
          ssh-keyscan github.com >> ~/.ssh/known_hosts
          git clone git@github.com:chaosfabric/homebrew-tap.git tap
          mkdir -p tap/Formula
          cp Formula-brainyz.rb tap/Formula/brainyz.rb
          cd tap
          git -c user.name='brainyz-release-bot' \
              -c user.email='41898282+github-actions[bot]@users.noreply.github.com' \
              commit -am "brainyz ${{ steps.meta.outputs.version }}" || {
            echo "No change; formula already at version ${{ steps.meta.outputs.version }}."
            exit 0
          }
          git push
```

```bash
#!/usr/bin/env bash
# Render tools/templates/scoop-brainyz.json.tmpl for a given release.
#
# Usage: render-scoop-manifest.sh <version> <tag> <sha256sums-path>
# See render-homebrew-formula.sh for the contract (identical shape).

set -euo pipefail

if [[ $# -ne 3 ]]; then
  echo "usage: $0 <version> <tag> <sha256sums-path>" >&2
  exit 1
fi

VERSION="$1"
TAG="$2"
SUMS="$3"

if [[ ! -f "$SUMS" ]]; then
  echo "error: SHA256SUMS file not found: $SUMS" >&2
  exit 1
fi

sha_for() {
  local file="$1"
  local sha
  sha=$(awk -v f="$file" '$2 == f { print $1; found=1 } END { exit !found }' "$SUMS") || {
    echo "error: missing checksum for $file in $SUMS" >&2
    return 1
  }
  printf '%s' "$sha"
}

SHA_WIN_X64=$(sha_for "brainz-win-x64.zip")

TMPL_DIR="$(cd "$(dirname "$0")/templates" && pwd)"
sed -e "s/{{VERSION}}/$VERSION/g" \
    -e "s/{{TAG}}/$TAG/g" \
    -e "s/{{SHA256_WIN_X64}}/$SHA_WIN_X64/g" \
    "$TMPL_DIR/scoop-brainyz.json.tmpl"
```

```yaml
        run: |
          LATEST=$(curl -fsSL https://api.github.com/repos/FavitoX/brainyz/releases/latest | jq -r .tag_name)
          echo "version=${LATEST#v}" >> "$GITHUB_OUTPUT"

        id: brew
        run: |

                | grep -oE 'version "[^"]+"' | head -n1 | cut -d'"' -f2 || true)
          echo "version=${VER:-missing}" >> "$GITHUB_OUTPUT"

      - name: Scoop bucket version
        id: scoop
        run: |
          VER=$(curl -fsSL https://raw.githubusercontent.com/chaosfabric/scoop-bucket/HEAD/bucket/brainyz.json \
                | jq -r .version 2>/dev/null || true)
          echo "version=${VER:-missing}" >> "$GITHUB_OUTPUT"

      - name: Winget manifest version
        id: winget
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # The folder name IS the version — list and take the largest by version-sort.
          VER=$(gh api repos/microsoft/winget-pkgs/contents/manifests/c/ChaosFabric/Brainyz \
                --jq '.[].name' 2>/dev/null | sort -V | tail -n1 || true)
          echo "version=${VER:-missing}" >> "$GITHUB_OUTPUT"

      - name: AUR version
        id: aur
        run: |
          VER=$(curl -fsSL 'https://aur.archlinux.org/rpc/?v=5&type=info&arg=brainyz-bin' \
                | jq -r '.results[0].Version // empty' \
                | sed 's/-[0-9]*$//' 2>/dev/null || true)
          echo "version=${VER:-missing}" >> "$GITHUB_OUTPUT"

      - name: Compare and open/update drift issue
        uses: actions/github-script@v8
        env:
          GH_VER:     ${{ steps.gh.outputs.version }}
          BREW_VER:   ${{ steps.brew.outputs.version }}
          SCOOP_VER:  ${{ steps.scoop.outputs.version }}
          WINGET_VER: ${{ steps.winget.outputs.version }}
          AUR_VER:    ${{ steps.aur.outputs.version }}
        with:
          script: |
            const latest = process.env.GH_VER;
            const channels = {
              Homebrew: process.env.BREW_VER,
              Scoop:    process.env.SCOOP_VER,
              Winget:   process.env.WINGET_VER,
              AUR:      process.env.AUR_VER,
            };
            const stale = Object.entries(channels).filter(([, v]) => v !== latest);
            if (stale.length === 0) {
              core.info(`All channels in sync at ${latest}`);
              return;
            }

            const title = `Packaging drift: latest release is ${latest}`;
            const body = [
              `GitHub Release: **${latest}**`,
              '',
              '| Channel | Version | Status |',
              '|---|---|---|',
              ...Object.entries(channels).map(([c, v]) =>
                `| ${c} | ${v} | ${v === latest ? 'OK' : 'STALE'} |`),
              '',
              'Detected by `.github/workflows/packaging.yml`.',
              'Re-run the release workflow or investigate channel-specific updater logs.',
            ].join('\n');

            const { data: issues } = await github.rest.issues.listForRepo({
              owner: context.repo.owner, repo: context.repo.repo,
              state: 'open', labels: 'packaging-drift', per_page: 100,
            });
            const existing = issues.find(i => i.title === title);
            if (existing) {
              await github.rest.issues.update({
                owner: context.repo.owner, repo: context.repo.repo,
                issue_number: existing.number, body,
              });
              core.info(`Updated existing drift issue #${existing.number}`);
            } else {
              const { data: issue } = await github.rest.issues.create({
                owner: context.repo.owner, repo: context.repo.repo,
                title, body, labels: ['packaging-drift'],
              });
              core.info(`Opened drift issue #${issue.number}`);
            }
```

````markdown
# Packaging

brainyz ships through four native package managers in addition to
[GitHub Releases](https://github.com/FavitoX/brainyz/releases). This
doc is the reference for how each channel works, how to bootstrap a new
release, and how to roll back a bad one.

## Channels

| Channel | Repository | Install command |
|---|---|---|
| Homebrew tap | [`chaosfabric/homebrew-tap`](https://github.com/chaosfabric/homebrew-tap) | `brew install chaosfabric/tap/brainyz` |
| Scoop bucket | [`chaosfabric/scoop-bucket`](https://github.com/chaosfabric/scoop-bucket) | `scoop bucket add chaosfabric https://github.com/chaosfabric/scoop-bucket; scoop install brainyz` |
| Winget | [`microsoft/winget-pkgs`](https://github.com/microsoft/winget-pkgs) (mirror at [`chaosfabric/winget-manifests`](https://github.com/chaosfabric/winget-manifests)) | `winget install ChaosFabric.Brainyz` |
| AUR | [`aur/brainyz-bin`](https://aur.archlinux.org/packages/brainyz-bin) (mirror at [`chaosfabric/aur-brainyz-bin`](https://github.com/chaosfabric/aur-brainyz-bin)) | `yay -S brainyz-bin` |

`SHA256SUMS.txt` published with each GitHub Release is the trust root
for all four channels; manifests pin to it directly.

## Release-time automation

On every stable tag (`vX.Y.Z` with no pre-release suffix), `.github/workflows/release.yml`:

1. Builds `brainz` for 4 RIDs (linux-x64, linux-arm64, osx-arm64, win-x64).
2. Runs the smoke-test per RID.
3. Generates `SHA256SUMS.txt` (`checksums` job).
4. Publishes the GitHub Release with archives + checksums.
5. **`update-homebrew-tap`** job: renders `Formula/brainyz.rb` from
   `tools/templates/homebrew-brainyz.rb.tmpl` + `tools/render-homebrew-formula.sh`,
   pushes to `chaosfabric/homebrew-tap`.
6. **`update-scoop-bucket`** job: renders `bucket/brainyz.json` similarly,
   pushes to `chaosfabric/scoop-bucket`.

Both updater jobs are `continue-on-error: true` and skip on
pre-release tags (anything with `-`). The drift-detection job in
`packaging.yml` opens an issue if any channel falls behind.

Winget and AUR are **not automated in v0.4**; they are bootstrapped
manually per the next section. v0.5+ will add `update-winget` and
`update-aur` jobs once the first-release manifests are proven.

## Per-release verification checklist

Run on clean machines (fresh container / VM / WSL distro) after each
stable release:

- [ ] **Homebrew (macOS arm64)**: `brew install chaosfabric/tap/brainyz`; then `brainz --version` and `bz --version` both print `brainyz <ver>`.
- [ ] **Homebrew (Linux x64)**: same, in a fresh Ubuntu container with Homebrew on Linux.
- [ ] **Scoop (Windows x64)**: `scoop install chaosfabric/brainyz`; `brainz --version` + `bz --version`.
- [ ] **Winget (Windows x64)**: `winget install ChaosFabric.Brainyz`; `brainz --version` + `bz --version`.
- [ ] **AUR (Arch x64)**: `yay -S brainyz-bin`; `brainz --version` + `bz --version` + `LD_DEBUG=libs brainz --version 2>&1 | grep libsql` finds the lib.
- [ ] Each channel: a minimal smoke cycle — `brainz init --here` in a tempdir → `brainz add note "smoke"` → `brainz list` returns a row.

## Rollback

If a release ships broken manifests to any channel:

| Channel | Action |
|---|---|
| Homebrew tap | `git revert <bad-commit> && git push` on `chaosfabric/homebrew-tap`. |
| Scoop bucket | `git revert <bad-commit> && git push` on `chaosfabric/scoop-bucket`. |
| Winget | PR to `microsoft/winget-pkgs` removing the offending version folder. |
| AUR | `git push` previous PKGBUILD (`git revert` + `git push` to the AUR remote). |

The GitHub Release itself is never edited — the underlying archives
remain available for manual install throughout. Users already on the
bad version downgrade with the package manager's standard
uninstall+install at the prior version.

## Future automation (v0.5+)

- **Winget auto-update**: CI job running `wingetcreate update --urls <new-url> --version <ver> ChaosFabric.Brainyz --submit --token $WINGET_PAT` on each stable tag. Needs a `WINGET_PAT` repo secret.
- **AUR auto-push**: CI job templating PKGBUILD + `.SRCINFO`, `git push`-ing to `ssh://aur@aur.archlinux.org/brainyz-bin.git` over an `AUR_SSH_KEY` secret.
- **Homebrew core submission**: after six consecutive stable releases with no regressions AND project meets
  [Acceptable Formulae](https://docs.brew.sh/Acceptable-Formulae) criteria.
- **nix flake**: when requested via issue.
````
